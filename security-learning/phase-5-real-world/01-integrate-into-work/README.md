# 01 融入日常测试 —— 安全左移

> 学了这么多，最终目标是**在你现有的测试工作中加入安全视角**。
> 不是要你变成全职安全工程师，而是成为**懂安全的测试工程师**。
> 这个组合在市场上非常稀缺，也非常有价值。

---

## 一、在日常测试中加入安全检查

### 1.1 每个功能测试时多做的"安全三问"

```
不需要额外时间，只需要在功能测试时多想三个问题：

1. "这个输入点能不能注入？"
   → 每个输入框、每个参数，顺手试一下 ' " < > ; |

2. "这个数据换个人能不能看到/改掉？"
   → 改一下请求中的ID、Token，看响应变不变

3. "这个流程能不能跳过/打乱？"
   → 直接访问后续步骤的URL，看服务端检不检查

花不了几分钟，但能发现大量安全问题。
```

### 1.2 不同功能类型的安全检查重点

```
[登录/注册]
  □ 暴力破解防护
  □ 用户名枚举
  □ 密码强度
  □ 传输加密
  □ Session管理

[搜索/查询]
  □ SQL注入
  □ XSS（搜索结果是否显示搜索词）
  □ 响应中是否返回过多数据

[表单提交（新建/编辑）]
  □ CSRF Token
  □ 输入验证（前端和后端都有？）
  □ 参数篡改（价格、角色、状态）
  □ 越权（改别人的数据）

[文件上传]
  □ 文件类型限制
  □ 文件大小限制
  □ 文件内容检测
  □ 上传路径安全

[列表/详情展示]
  □ 越权（改ID看别人数据）
  □ 分页限制（一次能取多少条）
  □ 敏感字段是否脱敏

[导出/下载]
  □ 导出权限控制
  □ 导出内容范围控制
  □ 下载路径遍历

[支付/交易]
  □ 金额篡改
  □ 竞态条件
  □ 订单状态篡改
  □ 优惠券/折扣逻辑
```

### 1.3 在测试计划中加入安全项

```
在你现有的测试用例模板中增加一列"安全检查"：

| 功能 | 功能测试 | 安全检查 |
|------|---------|---------|
| 用户登录 | 正确/错误密码、空值 | 暴力破解、枚举、SQL注入 |
| 修改资料 | 修改成功/失败 | 越权、CSRF、XSS |
| 搜索商品 | 搜索结果正确 | SQL注入、XSS |
| 下订单 | 下单成功/失败 | 价格篡改、越权、竞态 |
| 文件上传 | 正常上传 | 类型绕过、大小限制 |

不需要每次都做全面安全测试，但要有意识地检查。
```

## 二、向团队推广安全意识

```
[1] 从你发现的漏洞开始
  在内部分享会上演示一个你发现的安全问题
  现场演示 > 文档报告（冲击力更大）

[2] 给开发同事提供安全编码参考
  "这个接口需要加参数化查询"
  "这个Cookie需要设HttpOnly"
  → 具体、可操作的建议，而不是空洞的"注意安全"

[3] 推动安全测试流程化
  将安全检查项加入测试流程
  新功能上线前必须通过安全检查清单
  建立安全回归测试用例
```
