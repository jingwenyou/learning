# 01 网络基础 —— 数据在网络中怎么走的

> 学安全测试，第一件事不是学攻击，而是搞懂"数据是怎么从A到B的"。
> 你拦截、篡改、重放数据包——前提是你得知道包长什么样、怎么走的。

---

## 一、为什么测试工程师必须懂网络

你做功能测试时，点一个按钮，页面出了结果。你关心的是"结果对不对"。

做安全测试时，你关心的是：
- 这个请求里带了什么数据？（能不能被偷看）
- 这个数据经过了哪些节点？（能不能被中间人截获）
- 服务端怎么处理这个数据的？（能不能被篡改利用）

**网络基础就是帮你建立"数据流动"的全局视角。**

## 二、网络模型：不要背，要理解它解决了什么问题

### 2.1 一个类比：寄快递

你在网上买东西，卖家发货，快递到你手里。这个过程：

```
你下单（应用层）
  → 商品被打包（传输层：把数据切成包，编号，确保不丢）
    → 包裹贴上地址标签（网络层：IP地址，决定走哪条路）
      → 快递员骑车/开车送（数据链路层+物理层：实际传输）
```

网络模型就是这么回事——**每一层解决一个问题，层层封装，层层拆包**。

### 2.2 TCP/IP四层模型（实际用的）

| 层 | 解决什么问题 | 关键协议 | 你需要知道的 |
|----|------------|---------|------------|
| 应用层 | 数据的含义和格式 | HTTP, HTTPS, DNS, FTP | 安全测试的主战场 |
| 传输层 | 数据可靠到达 | TCP, UDP | TCP三次握手，端口的概念 |
| 网络层 | 数据走哪条路 | IP, ICMP | IP地址，路由 |
| 网络接口层 | 物理传输 | Ethernet, WiFi | 了解即可 |

### 2.3 你必须记住的核心概念

**IP地址** —— 网络中设备的"门牌号"
```
192.168.1.100      ← 内网地址（私有地址，局域网内用）
8.8.8.8            ← 公网地址（全球唯一）
127.0.0.1          ← 回环地址（指自己）

安全相关：
- 内网地址和公网地址的区别，直接关系到SSRF漏洞的理解
- 127.0.0.1 和 localhost 的含义，后面学SSRF时会反复碰到
```

**端口** —— 同一个IP上不同服务的"房间号"
```
一台服务器（一个IP）上可以跑很多服务：
  80   → HTTP（Web服务）
  443  → HTTPS（加密的Web服务）
  3306 → MySQL数据库
  22   → SSH远程登录
  8080 → 常见的开发/测试服务端口

安全相关：
- 端口扫描（Nmap）就是在探测目标开了哪些"房间"
- 数据库端口暴露在公网 = 重大安全风险
```

## 三、TCP：可靠传输的代价

### 3.1 三次握手（建立连接）

```
客户端                    服务端
  |                        |
  |------- SYN ---------->|   "我想连你"
  |                        |
  |<----- SYN+ACK --------|   "好的，我也准备好了"
  |                        |
  |------- ACK ---------->|   "确认，开始通信"
  |                        |
  [连接建立，可以传数据了]
```

**为什么是三次，不是两次？**

两次的问题：如果客户端的第一个SYN在网络中延迟了很久才到达服务端，
服务端会以为是新连接，白白分配资源等着。三次握手让双方都确认"对方还活着"。

**安全相关：**
- **SYN Flood攻击**：攻击者只发SYN不回ACK，让服务端一直等，耗尽资源
- 这是最经典的DDoS攻击之一，理解了握手过程就能理解这个攻击

### 3.2 四次挥手（断开连接）

```
客户端                    服务端
  |------- FIN ---------->|   "我说完了"
  |<------ ACK -----------|   "知道了，等我说完"
  |                        |   （服务端继续发送剩余数据）
  |<------ FIN -----------|   "我也说完了"
  |------- ACK ---------->|   "好的，拜拜"
```

**为什么断开要四次？** 因为TCP是全双工的——双方都能同时说话，所以每一方都要独立地说"我说完了"。

### 3.3 TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| 可靠性 | 保证到达、保证顺序 | 不保证 |
| 速度 | 慢（有握手、确认机制） | 快 |
| 场景 | 网页、文件传输、邮件 | 视频流、DNS查询、游戏 |
| 比喻 | 挂号信 | 扔纸飞机 |

**安全相关：** DNS用的UDP，所以DNS欺骗比TCP协议的攻击容易得多（没有握手验证）。

## 四、DNS：域名到IP的翻译官

### 4.1 基本流程

你在浏览器输入 `www.example.com`，发生了什么：

```
1. 浏览器检查自己的DNS缓存 → 没找到
2. 查操作系统的DNS缓存   → 没找到
3. 查本机hosts文件        → 没找到（/etc/hosts 或 C:\Windows\System32\drivers\etc\hosts）
4. 问本地DNS服务器（通常是路由器或ISP提供的）
5. 本地DNS服务器递归查询：
   根DNS服务器 → .com域服务器 → example.com的权威DNS服务器
6. 拿到IP：93.184.216.34
7. 浏览器用这个IP去建立TCP连接
```

### 4.2 为什么安全测试要懂DNS

```
场景1：DNS劫持
  你输入 bank.com，正常应该解析到银行的IP
  如果DNS被劫持，可能把你导向一个钓鱼网站的IP
  → 页面长得一模一样，你输入了账号密码

场景2：hosts文件在测试中的用处
  测试环境和生产环境用不同的IP
  通过修改hosts文件，可以把域名指向测试服务器
  安全测试时经常需要这样做

场景3：DNS重绑定攻击（SSRF进阶）
  第一次DNS解析返回正常IP（绕过检查）
  第二次解析返回内网IP（访问内部资源）
  → 这是绕过SSRF防护的常见手法
```

## 五、深入理解：一次完整的网络请求

你在浏览器输入 `https://www.example.com/login` 按回车，完整过程：

```
[1] DNS解析
    www.example.com → 93.184.216.34

[2] TCP三次握手
    你的电脑:随机端口 ←→ 93.184.216.34:443

[3] TLS握手（因为是HTTPS）
    协商加密算法 → 验证服务器证书 → 生成对称密钥
    （详见HTTP模块）

[4] 发送HTTP请求
    GET /login HTTP/1.1
    Host: www.example.com
    Cookie: session=abc123
    ...

[5] 服务器处理请求，返回响应
    HTTP/1.1 200 OK
    Content-Type: text/html
    Set-Cookie: session=xyz789
    ...
    <html>登录页面内容</html>

[6] 浏览器渲染页面

[7] 如果你点了"登录"按钮
    浏览器发送POST请求，带上你输入的用户名密码
    → 这个POST请求就是安全测试的重点关注对象
```

**安全测试的视角：**
- 第[1]步：DNS能不能被劫持？
- 第[3]步：证书合不合法？有没有用过时的TLS版本？
- 第[4]步：Cookie有没有设HttpOnly？Secure？
- 第[5]步：响应头有没有安全相关的Header？
- 第[7]步：密码是明文传输还是加密的？能不能被重放？

## 六、动手实践

### 实践1：用命令观察网络

```bash
# 查看本机IP
ip addr                        # Linux
ifconfig                       # macOS

# 查看DNS解析过程
nslookup www.baidu.com
dig www.baidu.com              # 更详细（Linux/macOS）

# 查看当前网络连接
ss -tunlp                     # Linux：查看所有监听的端口
netstat -an                   # 通用

# 查看路由
ip route                      # Linux
traceroute www.baidu.com      # 跟踪数据包经过的节点
```

### 实践2：抓包观察TCP握手

```bash
# 安装tcpdump（Linux）
apt install tcpdump

# 抓取访问百度时的TCP握手
# 在一个终端运行：
sudo tcpdump -i any host www.baidu.com -n -c 20

# 在另一个终端运行：
curl -I https://www.baidu.com

# 观察输出中的 SYN → SYN+ACK → ACK 过程
# 你会看到类似：
# Flags [S]      ← SYN
# Flags [S.]     ← SYN+ACK
# Flags [.]      ← ACK
```

### 实践3：观察DNS

```bash
# 查看本机DNS配置
cat /etc/resolv.conf

# 手动解析域名，观察完整过程
dig +trace www.example.com

# 修改hosts文件实验（改完记得恢复）
# 把 www.example.com 指向 127.0.0.1
echo "127.0.0.1 www.example.com" >> /etc/hosts
ping www.example.com    # 会发现ping的是127.0.0.1
# 恢复：删掉刚才加的那行
```
