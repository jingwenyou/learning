# 01 业务逻辑漏洞 —— 最难自动化检测的漏洞

> 扫描器能找到SQL注入和XSS，但找不到业务逻辑漏洞。
> 这类漏洞不是技术层面的缺陷，而是**业务流程的设计缺陷**。
> 只有**理解业务**的人才能发现——这正是测试工程师的优势。

---

## 一、什么是业务逻辑漏洞

```
技术漏洞：代码写错了（如SQL拼接、输出不编码）
逻辑漏洞：代码"正确"执行了，但业务流程有漏洞

每行代码都没有bug，但组合起来的业务流程有问题。
```

---

## 二、常见业务逻辑漏洞模式

### 2.1 流程跳过

```
正常流程：选商品 → 填地址 → 支付 → 完成
漏洞：    选商品 → 直接访问"完成"页面的URL → 跳过支付

测试方法：
  1. 走完正常流程，记录每一步的URL/请求
  2. 跳过中间步骤，直接请求后面步骤的接口
  3. 观察服务端是否检查前置步骤完成

真实案例：
  - 注册流程：跳过邮箱验证直接激活账号
  - 考试系统：直接访问成绩页面，跳过答题
  - 审批流程：跳过审批直接执行
```

### 2.2 参数篡改

```
[价格篡改]
  POST /api/order
  Body: {"product_id": 1, "quantity": 1, "price": 9999}
  修改：{"product_id": 1, "quantity": 1, "price": 0.01}
  如果服务端用前端传来的价格而不是查数据库 → 0.01元购买

[数量篡改]
  quantity: -1  → 负数数量，可能导致退款
  quantity: 0   → 零元购买
  quantity: 999999999  → 超大数量，可能整数溢出

[状态篡改]
  {"order_status": "pending"} → {"order_status": "completed"}
  {"user_role": "user"} → {"user_role": "admin"}

[优惠券/折扣]
  同一张优惠券使用两次
  多张优惠券叠加
  过期优惠券是否还能用
  修改折扣比例参数
```

### 2.3 竞态条件（Race Condition）

```
场景：余额100元，转账100元给朋友

正常：
  请求1 → 读余额100 → 扣款100 → 余额0 → 转账成功
  请求2 → 读余额0 → 余额不足 → 转账失败

竞态：
  同时发两个请求：
  请求1 → 读余额100              → 扣款100 → 余额0 → 转账成功
  请求2 → 读余额100（在扣款前！）→ 扣款100 → 余额-100 → 也成功了！
  → 花100元转了200元

测试方法：
  用Burp Intruder或Turbo Intruder同时发送多个相同请求
  观察是否出现了"不该出现"的状态（余额为负、多次领取等）

常见竞态场景：
  - 转账/提现
  - 领取优惠券/红包（同一张券领两次）
  - 签到/打卡（同一天签到两次）
  - 库存扣减（超卖）
  - 投票/点赞（刷票）
```

### 2.4 业务规则绕过

```
[密码重置绕过]
  正常：输入旧密码 → 才能设新密码
  测试：从请求中删掉旧密码字段 → 服务端是否仍然检查？

[验证码绕过]
  验证码无限次尝试（没有频率限制）
  验证码不过期（收到后第二天还能用）
  验证码可预测（4位数字+无限尝试=最多10000次）
  验证码在响应中返回（服务端返回了正确验证码给前端比对）

[邀请码/限量功能]
  邀请码是否可以复用
  限量商品是否能通过并发抢超
  限制一人一次的活动能不能换参数多次参与

[退款逻辑]
  购买 → 使用 → 退款 → 免费使用
  退款后赠品/积分是否被收回
```

## 三、检测方法论

```
核心原则：思考"如果我是一个贪便宜/想作弊的用户，我会怎么做"

[1] 理解业务流程
  画出完整的业务流程图
  标记每一步的前置条件和约束

[2] 挑战每一个约束
  "只能用一次" → 试试用两次
  "必须先A再B" → 试试直接B
  "价格是100" → 试试改成0
  "每天限3次" → 试试第4次
  "只能自己的" → 试试别人的

[3] 边界场景
  负数、零、超大值、空值
  同时操作（竞态）
  重复提交
  取消后再恢复

[4] 和PM/业务方确认
  你发现的"异常行为"是bug还是feature？
  有些看起来像漏洞的其实是设计如此
```
