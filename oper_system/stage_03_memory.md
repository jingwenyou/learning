# 阶段 3：内存

## 上节课回顾

上节课看到进程是程序的运行实例，内核用 `task_struct` 记录每个进程的状态。`task_struct` 里有一个字段 `mm`，指向这个进程的内存信息。**这节课问：程序用的地址和真实物理地址是什么关系？**

---

## 1. 为什么需要"虚拟地址"？

**大白话：**

你写 C 代码时，定义了一个变量 `int x = 42`，你可以打印它的地址，比如 `0x7ffee4b2a3c0`。但这个地址是"真实的物理地址"吗？

**不是。** 这是一个**虚拟地址**（Virtual Address）。操作系统给每个进程一种幻觉：好像自己独占了整个内存空间。

**为什么要这么做？**

假设没有虚拟地址：
- 进程 A 的变量住在物理地址 1000
- 进程 B 的变量住在物理地址 1000
- 结果：进程 B 可以直接读写进程 A 的内存 → 安全漏洞，系统崩溃

有了虚拟地址：
- 进程 A 的虚拟地址 1000 映射到物理地址 5000
- 进程 B 的虚拟地址 1000 映射到物理地址 8000
- 两个进程互相看不到对方 → 隔离

**设计决策：**

- **Problem**：多个进程需要同时使用内存，但不能互相干扰
- **Solution**：每个进程有独立的虚拟地址空间，内核管理"虚拟地址 → 物理地址"的映射
- **Tradeoffs**：多了一层翻译，有性能开销（用 TLB 缓解，见第 4 节）
- **Limitation**：物理内存有限，虚拟地址空间再大，最终能用的还是物理内存

---

## 2. 虚拟地址空间布局

**大白话：**

每个进程的虚拟地址空间被分成几个区域，各司其职：

```
高地址（内核空间，用户进程不能直接访问）
┌───────────────────────────────┐ 0xFFFFFFFFFFFFFFFF
│                               │
│         内核空间               │ ← 内核代码、数据住这里
│                               │
├───────────────────────────────┤ 0x0000800000000000（用户空间上限）
│                               │
│           栈（Stack）          │ ← 函数调用、局部变量，向下增长 ↓
│               ↓               │
│          （随机偏移 ASLR）      │
├───────────────────────────────┤
│                               │
│   内存映射区（mmap region）    │ ← 共享库、mmap 文件、匿名映射
│                               │
├───────────────────────────────┤
│               ↑               │
│           堆（Heap）           │ ← malloc/new，向上增长 ↑
│                               │
├───────────────────────────────┤ brk 指针
│          BSS 段                │ ← 未初始化的全局变量
├───────────────────────────────┤
│          数据段（Data）         │ ← 已初始化的全局变量
├───────────────────────────────┤
│          代码段（Text）         │ ← 程序的可执行代码
│                               │
└───────────────────────────────┘ 0x0000000000000000（NULL 在这里）
低地址
```

**每个区域的作用：**

| 区域 | 存放什么 | 特点 |
|------|---------|------|
| 代码段（Text） | 程序的机器指令 | 只读，可执行 |
| 数据段（Data） | 已初始化的全局变量 | 可读写 |
| BSS 段 | 未初始化的全局变量 | 程序启动时自动清零 |
| 堆（Heap） | `malloc` 分配的内存 | 向上增长，手动管理 |
| mmap 区域 | 共享库（libc 等）、文件映射 | 动态加载 |
| 栈（Stack） | 函数的局部变量、返回地址 | 向下增长，自动管理 |

**为什么堆往上长、栈往下长？**

历史原因。早期系统内存很小，把堆和栈放在地址空间两端向中间增长，可以灵活利用剩余空间。现代系统地址空间足够大，这个限制已不重要，但格局保留了下来。

## 3. 页表：地址翻译的机制

**大白话：**

虚拟地址和物理地址的映射关系，存在一个数据结构里，叫**页表**（Page Table）。

内存不是按字节来管理的，而是按**页**（Page）管理。每页大小通常是 **4KB**（4096 字节）。

```
虚拟地址翻译过程：

虚拟地址 0x7fff1234 5678
          ─────────┬────────────
                   │
          ┌────────▼────────┐
          │      MMU        │  ← 内存管理单元，硬件做翻译
          │  查这个进程的页表 │
          └────────┬────────┘

**MMU（Memory Management Unit）是什么？**

MMU 是 CPU 芯片里的一个硬件模块，专门负责把虚拟地址翻译成物理地址。你的程序只能访问虚拟地址，MMU 负责查页表找到对应的物理地址。

```
程序眼里：   0x7fff1234 5678  （虚拟地址）
MMU 翻译后： 0x0001 234 5678  （物理地址）
```

为什么是硬件做而不是软件做？因为每次内存访问都要翻译，软件太慢了，硬件集成在 CPU 里，延迟只有 1-2 个时钟周期。
                   │
          ┌────────▼────────┐
          │     页表         │  ← 存在内存里的一张"地图"
          │  虚拟页 → 物理页  │
          └────────┬────────┘
                   │
          物理地址 0x0001 5678
```

**为什么用 4KB 为单位，而不是按字节管理？**

按字节管理：4GB 内存需要 40 亿条映射记录，页表太大。
按 4KB 分页：4GB 内存只需要约 100 万条记录，可以接受。

**多级页表（x86-64 用 4 级）：**

64 位地址空间极大，一级页表会占用大量内存。所以用多级页表，只在实际用到的地址范围建立页表项：

```
虚拟地址（48 位有效位）拆分：
┌──────┬──────┬──────┬──────┬─────────────┐
│ L4   │ L3   │ L2   │ L1   │ 页内偏移     │
│ 9位  │ 9位  │ 9位  │ 9位  │  12位（4KB） │
└──────┴──────┴──────┴──────┴─────────────┘

查找过程：
CR3 寄存器 → L4 页目录 → L3 → L2 → L1 → 物理页帧
```

**设计决策：**

- **Problem**：直接用一级页表，64 位地址空间需要的页表太大（数 TB）
- **Solution**：多级页表，未使用的地址范围不分配子页表
- **Tradeoffs**：查找从 1 次内存访问变成 4 次，有延迟
- **Limitation**：通过 TLB 缓存来弥补这个开销（见下节）

## 4. TLB：页表的缓存

**大白话：**

每次访问内存，都要先查页表，但页表本身也在内存里。这样每次内存访问实际上要访问 5 次（4 次查页表 + 1 次读数据）？

太慢了。所以 CPU 里有一个硬件缓存：**TLB**（Translation Lookaside Buffer，翻译后援缓冲区）。

```
CPU 访问虚拟地址：

TLB 命中（快，1 个时钟周期）：
虚拟地址 → TLB 查到了 → 物理地址 → 读数据

TLB 未命中（慢，需要走页表）：
虚拟地址 → TLB 没找到 → 走多级页表（4 次内存访问）→ 物理地址 → 读数据
                                                                    ↓
                                                          把结果存入 TLB
```

**典型数值：**

| 操作 | 典型耗时 |
|------|---------|
| TLB 命中 | 0.5-1 ns（约 1-2 个时钟周期） |
| TLB 未命中（页表查找） | 10-50 ns（4 次内存访问） |
| L1 cache hit | 1 ns |
| 主存访问 | 100 ns |

TLB 通常只有 64-1024 条缓存，但由于程序的局部性（反复访问同一段地址），命中率通常在 99% 以上。

**什么时候 TLB 会失效？**

进程切换时，换了一个进程，它的页表完全不同，老的 TLB 缓存无效，需要 flush（清空）。这就是进程切换有性能开销的原因之一。

## 5. 内存分配：brk 和 mmap

**大白话：**

你调用 `malloc(1024)` 请求 1KB 内存，这块内存从哪来？

用户态的 `malloc` 最终会向内核申请内存，有两种方式：

**方式 1：brk（调整堆的大小）**

`brk` 是堆顶的指针。调高 `brk` 就是扩大堆。

```
malloc 小内存（通常 < 128KB）：
  glibc 的 malloc → 检查自己维护的空闲列表
                  → 没有 → 调用 brk() 系统调用向内核要更多内存
                  → 内核把 brk 往上移
                  → glibc 把多出来的内存切一块给你
```

**方式 2：mmap（内存映射）**

大块内存用 `mmap` 分配，分配在 mmap 区域，释放时（`munmap`）直接还给内核。

```
malloc 大内存（通常 >= 128KB）：
  glibc 的 malloc → 调用 mmap() 系统调用
                  → 内核分配一块独立的匿名映射区域
                  → 返回地址给 malloc
  free 时 → 调用 munmap()，立刻归还内核
```

**为什么 free 之后内存不会立刻还给内核？**

`free` 只是告诉 glibc 这块内存可以复用了，glibc 放进自己的空闲列表。下次 `malloc` 先从空闲列表里找，减少系统调用次数。只有当 glibc 认为有太多空闲堆内存时，才会调用 `brk` 降低堆顶，把内存还给内核。

## 6. 内核内存管理：slab 分配器

**大白话：**

内核自己也需要频繁分配小对象（比如每创建一个进程要分配一个 `task_struct`，大约 8KB）。如果每次都走页级别的分配器（每次至少分配 4KB，还要初始化），效率很低。

**slab 分配器** 的思路：

```
slab 分配器预先分配一批同类型对象：

┌──────────────────────────────────────────┐
│  task_struct 的 slab 缓存                 │
│  ┌────────┐ ┌────────┐ ┌────────┐       │
│  │task #1 │ │task #2 │ │task #3 │ ....  │
│  │（空闲） │ │（使用中）│ │（空闲） │       │
│  └────────┘ └────────┘ └────────┘       │
└──────────────────────────────────────────┘

分配：从空闲对象中拿一个（近乎 O(1)，不用初始化）
释放：标记为空闲，放回缓存（不还给内核，等待复用）
```

**设计决策：**

- **Problem**：频繁分配/释放固定大小的内核对象，通用分配器效率低
- **Solution**：每种类型对象维护专属缓存，对象在缓存里循环利用
- **Tradeoffs**：内存可能提前占用（即使没在用，slab 也可能持有内存）
- **Limitation**：对象大小固定，不适合变长数据

> **slub** 是 slab 的简化版，是现在 Linux 的默认实现（slab 太复杂了，slub 在保持性能的同时大幅简化了代码）。

## 7. 破坏性实验

> **环境要求：** Linux 系统（Ubuntu/Debian/CentOS 等）。

### 实验 1：观察进程的虚拟地址空间

```bash
# 观察当前 shell 的内存映射（所有虚拟地址区域）
cat /proc/$$/maps

# 输出每一行的格式：
# 地址范围                权限 偏移 设备 inode 文件路径
# 7f8a1234b000-7f8a1234c000 r-xp 00000000 fd:01 12345 /usr/lib/libc.so
```

**验证标准：**

| 你应该看到 | 说明 |
|-----------|------|
| 权限为 `r-xp` 的行（有文件路径） | 代码段，可读可执行 |
| 权限为 `rw-p` 的行（有文件路径） | 数据段，可读写 |
| 权限为 `rw-p` 的行（无文件路径） | 堆或栈（匿名内存） |
| 地址最高的 `[stack]` 行 | 栈区 |

### 实验 2：用 C 程序观察各段的地址

```bash
cat > /tmp/test_mem.c << 'EOF'
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int global_init = 42;        // 数据段（已初始化全局变量）
int global_uninit;           // BSS 段（未初始化全局变量）

int main() {
    int local = 0;           // 栈
    void *heap = malloc(64); // 堆

    printf("代码段（main 函数）:       %p\n", (void*)main);
    printf("数据段（global_init）:     %p\n", (void*)&global_init);
    printf("BSS 段（global_uninit）:   %p\n", (void*)&global_uninit);
    printf("栈（local 变量）:          %p\n", (void*)&local);
    printf("堆（malloc）:             %p\n", heap);

    free(heap);
    return 0;
}
EOF
gcc /tmp/test_mem.c -o /tmp/test_mem && /tmp/test_mem
```

**验证标准：**
- 代码段地址最低（小数字）
- 堆地址比数据段高
- 栈地址最高（大数字，靠近 `0x7fff...`）
- 地址间的大小关系符合上面的布局图

### 实验 3：触发 Segfault — 访问 NULL 指针

```bash
cat > /tmp/test_segfault.c << 'EOF'
#include <stdio.h>

int main() {
    int *p = NULL;
    printf("准备访问 NULL 指针...\n");
    *p = 42;  // 这行会触发 SIGSEGV
    printf("这行不会被执行\n");
    return 0;
}
EOF
gcc /tmp/test_segfault.c -o /tmp/test_segfault && /tmp/test_segfault
```

**验证标准：**
- 打印 "准备访问 NULL 指针..." 后程序终止
- 出现 `Segmentation fault (core dumped)` 或类似提示
- 原因：地址 0x0（NULL）没有映射到任何物理内存，MMU 触发缺页异常，内核发送 SIGSEGV 信号给进程

### 实验 4：观察 TLB miss 的性能影响

```bash
# 顺序访问内存（对 TLB 友好，连续地址局部性好）
time python3 -c "
import ctypes
size = 100 * 1024 * 1024  # 100MB
buf = (ctypes.c_byte * size)()
for i in range(0, size, 4096):  # 每 4KB 访问一次（一页一次）
    buf[i] = 1
print('顺序访问完成')
"

# 随机访问内存（TLB 不友好，随机地址导致大量 TLB miss）
time python3 -c "
import ctypes, random
size = 100 * 1024 * 1024  # 100MB
buf = (ctypes.c_byte * size)()
pages = list(range(0, size, 4096))
random.shuffle(pages)
for i in pages:
    buf[i] = 1
print('随机访问完成')
"
```

**验证标准：**
- 随机访问比顺序访问慢（通常慢 2-5 倍）
- 原因：随机访问导致大量 TLB miss，每次都要走 4 级页表查找

### 实验 5：观察内核 slab 使用情况

```bash
# 查看 slab 缓存统计
cat /proc/slabinfo | head -20

# 或用更友好的格式
sudo slabtop -o | head -20
```

**验证标准：**
- 能看到 `task_struct`、`mm_struct`、`files_cache` 等内核对象
- 每个缓存有 active_objs（使用中）和 num_objs（总量）两个计数

**/proc/slabinfo 怎么看？**

原始格式比较难读，先用 `slabtop` 看得更清楚：
```bash
sudo slabtop -o
```

常见列的含义：

| 列 | 含义 |
|----|------|
| `active_objs` | 正在使用的对象数量 |
| `num_objs` | 总共分配的对象数量 |
| `obj_size` | 每个对象的大小（字节） |
| `objperslab` | 每个 slab 页能放多少个对象 |
| `pagesperslab` | 每个 slab 占用多少页（通常 1 或更多） |

**实例解释**：
```
task_struct  active=45  num_objs=60  obj_size=4224
```
意思是：系统里当前有 45 个进程在运行，内核预分配了 60 个 task_struct 对象槽位，每个对象 4224 字节（约 4KB）。

**为什么 num_objs 比 active_objs 多？**

slab 分配器预分配了一批对象备用。进程销毁时，task_struct 不会还给内存，而是标记为空闲放回 slab 缓存，下次创建新进程时直接复用，省去分配和初始化的开销。

```
阶段 2（进程）
    │
    └── task_struct.mm（指向内存管理结构）
                  │
                  ▼
              内存（本节）
                  │
    ┌─────────────┼─────────────┐
    │             │             │
    ▼             ▼             ▼
虚拟地址空间     页表机制        内存分配
（布局）      （虚拟→物理）
    │             │             │
    │             ▼             ▼
    │            TLB        brk / mmap
    │          （缓存）
    │
    ▼
代码段、数据段、BSS、堆、栈
    │
    └── → 阶段 4（系统调用）
              bind/listen/recv 都要分配内存
    └── → 阶段 5（中断）
              缺页中断：访问未映射地址时触发
```

## 9. 调试场景

**"OOM Killer 把我的进程杀了，怎么回事？"**

```bash
# 查看内核日志里的 OOM 信息
dmesg | grep -i "oom"
dmesg | grep -i "killed process"

# 查看当前内存使用情况
free -h

# 查看进程内存使用（按 RES 排序）
ps aux --sort=-%mem | head -20
```

**"程序 malloc 了很多内存，但 free 之后 RSS 没降，内存泄漏了吗？"**

不一定是泄漏。glibc 的 `free` 把内存放回空闲列表，不立刻还给内核，所以 RSS 不降是正常的。
真正的内存泄漏是：`malloc` 了但**没有** `free`，导致空闲列表外的堆一直增长。

```bash
# 用 valgrind 检查内存泄漏
valgrind --leak-check=full ./your_program

# 观察进程内存随时间变化（如果一直涨才是泄漏）
watch -n 1 "cat /proc/<PID>/status | grep VmRSS"
```

**"程序崩溃出现 Segfault，怎么定位到哪一行？"**

```bash
# 编译时加 -g（保留调试符号）
gcc -g -o program program.c

# 运行并获取 core dump
ulimit -c unlimited
./program  # 崩溃后生成 core 文件

# 用 gdb 分析 core dump
gdb ./program core
# 在 gdb 里：
# bt    -- 查看调用栈（找到崩溃的那一行）
```

## 10. 常见问题 Q&A

**Q: 执行 `ulimit -c` 后，core dump 文件会在哪儿？**

A: 取决于两个因素：

1. **文件大小限制**：`ulimit -c` 的值决定 core 文件最大多大
   ```bash
   ulimit -c          # 查看当前值（0 = 禁用）
   ulimit -c unlimited  # 启用，不限制大小
   ```

2. **core_pattern 配置**：决定文件名和位置
   ```bash
   cat /proc/sys/kernel/core_pattern
   ```
   - 输出 `core`：保存在进程当前工作目录，文件名 `core`
   - 输出 `|/usr/share/apport/...`：通过 apport 管道处理（Ubuntu 默认，容器中可能不生成文件）
   - 输出 `/var/crash/core-%e-%p-%t`：保存在 `/var/crash/`

**Q: 程序崩溃了（Segmentation fault），但没看到 core 文件？**

A: 按以下顺序排查：

| 检查项 | 命令 | 解决方法 |
|--------|------|----------|
| ulimit 为 0 | `ulimit -c` | `ulimit -c unlimited` |
| core_pattern 是管道 | `cat /proc/sys/kernel/core_pattern` 第一位是 `\|` | `echo "core" > /proc/sys/kernel/core_pattern` |
| 容器环境 | apport 在容器中可能不生成 | 改用 `echo "core" > /proc/sys/kernel/core_pattern` |
| 磁盘空间 | `df -h .` | 清理磁盘 |
| 权限 | `ls -la .` | 确保可写 |

**排查命令**：
```bash
# 完整排查流程
ulimit -c unlimited
cat /proc/sys/kernel/core_pattern   # 如果是 | 开头，改掉
echo "core" > /proc/sys/kernel/core_pattern  # 改成本地文件模式
./test_segfault
ls -la core*   # 应该在当前目录找到 core 文件
```

**Q: MMU 是什么？**

A: **MMU（Memory Management Unit，内存管理单元）** 是 CPU 芯片里的硬件模块，专门负责把程序使用的**虚拟地址**翻译成**物理地址**。

```
程序只能看到虚拟地址，比如 0x7fff1234
MMU 查页表翻译成物理地址，比如 0x0001abcd
```

为什么是硬件做而不是软件做？因为每次内存访问都要翻译（读一次数据要查好几次页表），软件来做太慢了，MMU 硬件集成在 CPU 里，只需要 1-2 个时钟周期。

**Q: slab 是什么？**

A: **slab 分配器**是内核用来高效分配小对象的一种机制。

内核经常要分配固定大小的对象，比如：
- 创建进程要分配 `task_struct`（约 4KB）
- 打开文件要分配 `files_struct`（约 4KB）

如果每次都从页分配器要 4KB，初始化还要清零，效率很低。slab 的做法是：
1. 预先分配一大块内存（一段或几段页）
2. 切成固定大小的小块对象
3. 分配时直接拿一块（不需要初始化，因为 slab 创建时已经初始化好了）
4. 释放时标记为空闲，不还给内核，下次直接复用

**slub** 是 slab 的改进版，是现在 Linux 的默认实现，代码更简单但性能一样好。

**Q: /proc/slabinfo 怎么看不懂？**

A: 原始格式确实不友好，建议用 `slabtop`：
```bash
sudo slabtop -o
```

常见列的含义：

| 列 | 含义 |
|----|------|
| `active_objs` | 正在使用的对象数量 |
| `num_objs` | 预分配的总对象数量 |
| `obj_size` | 每个对象的大小（字节） |
| `objperslab` | 每个 slab 页能放多少个对象 |

**实例解释**：
```
task_struct  active=45  num_objs=60  obj_size=4224
```
意思是：系统当前有 45 个进程在运行，内核预分配了 60 个 task_struct 槽位，每个 4224 字节。

**为什么 num_objs 比 active_objs 多？** 预分配备用，进程结束时对象放回 slab 缓存而不是释放内存，下次创建新进程时直接复用，省去分配和初始化的开销。
