# AI 闭环测试 — 系统化学习计划（修订版）

> 背景：测试工程师，测安全/网络/密码一体机产品（带Web管理页面），用禅道，会用pytest
> 目标：从提示词使用者 → 能用AI贯穿测试全流程
> 原则：每个阶段产出可用成果，嵌入现有工作流，不另起炉灶

用户反馈 → 质量度量 → 过程改进 ──→ 反哺下一轮需求评审（闭环）
 ↑ AI聚类   ↑ AI趋势   ↑ AI建议

─────────────────────────────────────────────────────────────────────────────────────
                              AI效果度量（持续跟踪）

你的产品场景贯穿每个环节：
- Web管理页面 → UI测试、接口测试

## 阶段一：提示词驱动测试（第1-2周）

### 目标：用提示词覆盖测试核心环节，并学会判断AI产出质量

### 1.1 ★ 质量左移：AI辅助需求评审（在编码之前发现问题）

**为什么重要**：需求阶段发现一个问题的修复成本是测试阶段的1/10。左移不是多做事，是把发现问题的时间点提前。

**场景一：需求文档缺陷检测**
```
你是一名资深测试架构师，擅长在需求阶段发现隐患。

以下是一份产品需求文档：
"""
[粘贴需求文档]
"""

请从测试角度审查，找出：
1. 模糊描述（如"支持大量用户" — 多少算大量？）
2. 遗漏场景（只描述了正常流程，没提异常/边界/回退）
3. 矛盾点（前后描述不一致的地方）
4. 安全风险（需求本身是否引入安全隐患，如明文传输、默认弱密码）
5. 可测试性问题（无法验证的需求，如"系统应足够快"）
6. 对现有功能的潜在影响（改了A是否会影响B？）

对每个问题标注严重程度，并给出改进建议。
```

**场景二：设计方案可测试性评审**
```
以下是[功能名]的技术设计方案：
"""
[粘贴设计文档/接口设计]
"""

请评估：
1. 各接口是否可独立测试？（是否有隐式依赖）
2. 错误码设计是否完整？（能否通过错误码定位问题）
3. 日志设计是否支持问题排查？（关键操作是否有审计日志）
4. 是否有可观测性设计？（状态查询、健康检查接口）
5. 降级/容错方案是否可验证？

这些直接影响你后续测试的效率和问题定位速度。
```

**实操方式**：每次参加需求评审前，先把需求文档发给AI做预审，带着AI找到的问题去评审会。

### 1.2 需求分析 → 测试点提取

**输入**：产品需求文档（如：一体机新增VPN策略配置功能）
**AI做什么**：提取功能点、识别安全风险点、标注边界条件
**输出**：结构化测试点清单

**提示词模板**：
```
你是一名资深安全产品测试工程师。

以下是一个安全/网络产品的需求：
"""
[粘贴需求内容]
"""

请按以下结构提取测试点：
1. 功能测试点（正常流程）
2. 异常测试点（错误输入、边界值、并发）
3. 安全测试点（权限、注入、越权、加密）
4. 兼容性测试点（浏览器、分辨率 — 针对Web管理页面）
5. 关联影响（该功能变更可能影响的其他模块）

对每个测试点标注：优先级（P0-P3）、测试类型（手工/可自动化）
```

**练习**：拿你当前正在测的一个功能，用这个模板，对比你自己的测试点。

### 1.3 测试用例生成

**输入**：测试点 + 业务规则
**AI做什么**：按等价类、边界值、错误推测法生成用例
**输出**：可导入禅道的用例表格

**提示词模板**：
```
基于以下测试点，生成详细测试用例：

功能：[如：Web管理页面的密码策略配置]
测试点：[如：密码长度限制8-32位]

要求：
1. 用表格格式：用例编号 | 用例标题 | 前置条件 | 测试步骤 | 预期结果 | 优先级
2. 覆盖：有效等价类、无效等价类、边界值（7位/8位/32位/33位）、特殊字符
3. 针对安全产品，额外考虑：SQL注入、XSS、暴力破解、会话管理
4. 格式兼容禅道导入（CSV）
```

### 1.4 测试数据构造

**针对你的产品场景**：
- IP地址数据（合法/非法/边界：0.0.0.0、255.255.255.255、IPv6）
- 密码数据（不同强度、特殊字符、超长、空密码）
- 网络配置数据（端口范围、VLAN ID、子网掩码）
- 证书/密钥测试数据

**提示词模板**：
```
为网络安全产品生成测试数据，CSV格式：

场景：[如：防火墙规则配置]
字段：源IP、目的IP、端口、协议、动作
要求：
- 20条正常数据
- 20条异常数据（无效IP、端口越界、协议不支持）
- 10条边界数据
- 5条安全测试数据（注入尝试）
```

### 1.5 Bug描述优化

**提示词模板**：
```
我发现了一个bug，原始描述如下：
"""
[你的粗略描述]
"""

产品类型：安全/网络一体机，Web管理页面
请帮我重写为标准bug报告：
1. 标题（一句话概括）
2. 环境（设备型号、固件版本、浏览器）
3. 复现步骤（编号列出，每步一个操作）
4. 实际结果
5. 预期结果
6. 严重程度建议（致命/严重/一般/轻微）+ 理由
7. 是否涉及安全风险
```

### 1.6 ★ AI产出质量审查（核心能力）

**AI会犯的典型错误**：
| 问题 | 示例 | 怎么发现 |
|------|------|----------|
| 遗漏安全场景 | 只测正常登录，没测暴力破解/会话劫持 | 对照OWASP Top 10检查 |
| 场景不切实际 | 生成的IP地址格式不对、端口范围错误 | 用你的产品知识判断 |
| 用例重复 | 换了措辞但本质相同的用例 | 去重检查 |
| 缺少前置条件 | 步骤写了"配置策略"但没说先要登录 | 按步骤心里走一遍 |
| 不了解你的产品 | 建议测"支付功能"（你们没有） | 检查是否匹配你的产品 |

**审查checklist**：
- [ ] 每个功能的正常/异常/边界/安全场景都覆盖了？
- [ ] 用例步骤能在真实设备上执行？（不是AI臆想的操作）
- [ ] 安全相关用例是否覆盖了权限、加密、注入、会话？
- [ ] 数据是否符合你产品的实际字段约束？
- [ ] 有没有AI编造的不存在的功能或页面？

### 1.7 AI辅助测试计划与策略

**输入**：项目背景、版本范围、资源约束
**AI做什么**：生成测试策略、风险评估、优先级排序
**输出**：结构化测试计划

**提示词模板**：
```
你是一名资深测试经理，正在规划一个安全/网络产品版本的测试策略。

项目信息：
"""
[版本号、涉及模块、开发周期、测试资源]
"""

请输出：
1. 测试范围（按模块列出，标注新增/变更/回归）
2. 风险评估矩阵（模块 × 风险等级，标注理由）
3. 测试优先级排序（基于风险和业务影响）
4. 资源分配建议（哪些适合自动化、哪些必须手工）
5. 退出标准（什么条件下可以发版）
```

**深度用法**：把上一版本的bug分布数据一起给AI，让它基于历史数据做风险预测。

### 1.8 AI引导探索性测试

**为什么需要**：脚本化测试覆盖已知场景，但安全产品往往在"意料之外"的路径上出问题。

**提示词模板**：
```
我正在对以下功能做探索性测试：

功能：[如：一体机的证书管理模块]
已有用例覆盖：[简述已有用例范围]

请作为一名攻击者视角的测试专家，给我：
1. 5个"如果我是攻击者会怎么尝试"的测试思路
2. 5个容易被忽略的边缘交互场景（如：证书过期时同时修改网络配置）
3. 3个跨模块联动的异常场景
4. 每个思路标注：探索方向、观察什么、可能发现什么类型的缺陷
```

**练习**：每次测试前花5分钟让AI给探索性方向，记录实际发现了多少额外bug。

### 1.9 ★ 提示词迭代方法

**三轮迭代法**：
```
第1轮：粗提示 → 看AI理解对不对，输出格式对不对
第2轮：补上下文 → 加产品背景、业务规则、具体约束
第3轮：纠错优化 → 针对AI遗漏/错误的点，追问补充

每轮记录：哪些提示词改动带来了明显改善 → 形成你自己的提示词库
```

- [ ] 每个模板经过3轮迭代优化，记录了改进过程
- [ ] 一份AI产出质量审查checklist
- [ ] 至少3个真实功能的AI生成用例，经过你审查修正后可导入禅道
- [ ] 至少1次用AI预审需求文档，并在评审会上验证AI发现的问题

## 阶段二：用AI工具写代码（第3-4周）

### 目标：借助Claude Code / Cursor，让AI帮你写pytest脚本

> **关键转变**：不是你学会写Python，而是你学会"指挥AI写Python"
> 你已经有pytest基础（看过、简化过别人的框架），这是优势

### 2.1 环境准备
- 安装Claude Code 或 Cursor（AI编程助手）
- 把你现有的pytest框架准备好
- **不需要从零学Python语法**，用AI写、你来审查和运行

### 2.2 AI帮你写接口测试（你的Web管理页面）

**工作流**：
```
你的操作：打开产品Web页面 → F12看网络请求 → 复制curl/请求信息
    ↓
发给AI：把这个请求写成pytest测试用例，放进我现有的框架里
    ↓
AI输出：pytest代码
    ↓
你的操作：跑一下 → 看结果 → 不对就把报错发给AI修
```

**练习场景**（贴合你的产品）：
- 登录接口测试（正常/错误密码/账号锁定/会话超时）
- 网络配置接口（添加/修改/删除防火墙规则）
- 密码策略接口（设置/查询/违规检测）
- 权限接口（不同角色访问同一接口）

### 2.3 AI帮你写Web UI测试

**工作流**：
```
你的操作：截图 或 描述操作步骤
    ↓
发给AI：用Playwright写这个操作的自动化测试
    ↓
AI输出：Playwright代码
    ↓
你的操作：运行 → 看结果 → 修
```

### 2.4 AI辅助静态分析与代码评审

**左移延伸**：不只测运行时行为，在代码层面提前发现问题。

**场景一：让AI审查开发提交的代码变更**
```
以下是开发本次提交的代码变更（diff）：
"""
[粘贴git diff或代码片段]
"""

产品类型：安全/网络一体机
请从测试角度审查：
1. 是否有明显的逻辑错误？
2. 是否有安全隐患？（硬编码密钥、SQL拼接、未校验输入、权限检查遗漏）
3. 错误处理是否完整？（异常情况是否会导致信息泄露或服务崩溃）
4. 这次变更可能影响哪些现有功能？（回归风险评估）
5. 建议针对这次变更补充哪些测试用例？
```

**场景二：配置文件安全检查**（贴合你的一体机产品）
```
以下是产品的[防火墙规则/网络配置/证书配置]：
"""
[粘贴配置内容]
"""

请检查：
1. 是否有不安全的默认配置？（默认开放端口、弱加密算法）
2. 规则之间是否有冲突或遮蔽？（前面的规则让后面的永远不生效）
3. 是否有合规性问题？（如：使用了已废弃的加密套件）
```

**实操方式**：每次版本提测时，把关键代码变更过一遍AI审查，重点看安全相关变更。

### 2.5 AI写安全专项测试脚本

**为什么单独列**：你的产品是安全产品，安全测试不是附加项而是核心。

**场景一：接口安全扫描**
```
我的产品是一个安全/网络一体机，有Web管理页面。
以下是登录接口的请求信息：
"""
[粘贴curl或请求详情]
"""

请用pytest写一组安全测试：
1. SQL注入测试（用户名/密码字段，至少10种payload）
2. XSS测试（所有输入字段，存储型+反射型）
3. 暴力破解模拟（连续错误登录，验证是否触发锁定）
4. 会话安全测试（token过期、重放、并发登录）
5. 越权测试（普通用户token访问管理员接口）

输出可运行的pytest代码，每个测试标注它在检测什么漏洞。
```

**场景二：密码/加密合规性测试**
```
请用pytest写密码策略合规性测试：
产品要求：[如：符合GM/T 0054标准]

测试项：
1. 密钥长度是否符合标准（SM2/SM3/SM4）
2. 加解密结果正确性（用已知测试向量验证）
3. 随机数质量测试（重复性检测）
4. 证书链验证（自签名、过期、吊销）
```

### 2.6 AI写性能测试脚本

**场景**：你的一体机在高并发下的表现。

**工作流**：
```
你的操作：确定性能关注点（如：100用户同时登录、批量下发防火墙规则）
    ↓
发给AI：用locust/pytest写性能测试脚本
    ↓
AI输出：可运行的性能测试代码
    ↓
你的操作：运行 → 收集结果 → 让AI分析瓶颈
```

**提示词模板**：
```
用locust写一个性能测试脚本：

目标：Web管理页面的[具体接口]
场景：
- 50个并发用户持续请求60秒
- 每个用户：登录 → 查询配置 → 修改一条规则 → 登出
- 记录：响应时间、吞吐量、错误率

额外要求：
- 参数化用户账号
- 设置合理的思考时间（think time）
- 输出结果包含P50/P90/P99延迟
```

### 2.7 你需要掌握的最小Python知识

- **其他全让AI写**

## 阶段三：AI + 禅道工作流打通（第5周）

### 目标：让AI嵌入你的禅道工作流，减少重复劳动

### 3.1 禅道数据导出 → AI分析

**实操**：
```
从禅道导出bug列表（CSV/Excel）
    ↓
发给AI分析：
- 哪些模块bug最多？
- 最近版本质量趋势如何？
- 高频bug类型是什么？
- 哪些bug反复出现（回归bug）？
    ↓
输出：质量分析报告，可直接发给领导
```

### 3.2 测试报告自动摘要

**实操**：
```
pytest运行完 → 生成HTML/JSON报告
    ↓
把结果发给AI：
- 总结通过率
- 列出失败用例及可能原因
- 标注需要关注的模块
    ↓
输出：一段文字摘要
```

### 3.3 禅道用例批量生成

**实操**：
```
需求文档 → AI生成用例（阶段一的提示词）
    ↓
AI直接输出禅道可导入的CSV格式
    ↓
导入禅道 → 分配执行
```

### 3.4 ★ 质量右移：生产数据反哺测试

**为什么重要**：测试环境永远无法100%模拟生产。右移不是"上线后再测"，而是从生产中获取信息来改进测试。

**场景一：生产日志分析 → 发现测试盲区**
```
以下是产品一体机最近一周的运维日志/告警记录：
"""
[粘贴日志片段或告警摘要]
"""

请分析：
1. 出现了哪些异常？按频率和严重程度排序
2. 哪些异常是测试环境中可以复现的？
3. 哪些场景是当前测试用例没有覆盖的？（测试盲区）
4. 建议补充哪些测试用例？
5. 是否有安全相关告警需要重点关注？
```

**场景二：故障复盘 → 改进测试策略**
```
以下是一次线上故障的复盘记录：
"""
[故障描述、根因、影响范围、修复方案]
"""

请分析：
1. 这个故障能否在测试阶段被发现？如果能，缺了什么测试？
2. 需要补充的测试用例（具体到步骤级）
3. 现有测试流程中哪个环节出了问题？（遗漏/覆盖不足/环境差异）
4. 类似故障模式还可能出现在哪些模块？（举一反三）
5. 建议的流程改进项
```

**场景三：用户反馈聚类分析**
```
以下是最近收集到的客户反馈/技术支持工单：
"""
[粘贴反馈内容]
"""

请归类分析：
1. 按问题类型聚类（功能缺陷/性能/易用性/安全/兼容性）
2. 高频问题TOP 5
3. 哪些反映了测试遗漏？建议补充什么测试？
4. 哪些反映了需求理解偏差？建议反馈给产品经理
```

**实操方式**：每个版本上线后1-2周，收集生产数据做一次分析，把发现反哺到下个版本的测试策略中。这就是闭环。

### 3.5 缺陷预测与风险分析

**输入**：禅道历史bug数据 + 当前版本变更范围
**AI做什么**：识别高风险模块、预测缺陷密集区域
**输出**：风险热力图 + 回归测试优先级

**提示词模板**：
```
以下是过去3个版本的bug数据（CSV）：
"""
[粘贴或附加数据]
"""

当前版本变更了以下模块：[模块列表]

请分析：
1. 哪些模块历史缺陷密度最高？（按模块统计bug数/严重程度）
2. 当前变更模块中，哪些与高缺陷模块有依赖关系？
3. 基于历史规律，预测本版本最可能出问题的3个区域
4. 给出回归测试优先级排序建议
5. 有没有反复出现的"同类bug"模式？（如：每次改网络模块都会影响证书）
```

### 3.6 测试覆盖度评估

**实操**：
```

"""
[需求-用例映射表]
"""

请评估：
1. 哪些需求的用例覆盖明显不足？（给出判断依据）
2. 正常/异常/边界/安全四个维度，哪个维度普遍偏弱？
3. 建议补充哪些场景？按优先级排列
```

**价值**：从"测了多少"升级到"测得够不够"，这是测试工程师进阶的关键能力。

## 阶段四：工具优先 + 按需API（第6-8周）

### 目标：判断什么时候工具够用、什么时候需要API，把重复操作自动化

> 2026年的现实：Claude Code等AI工具已经能直接完成大量任务，不是所有场景都需要写API脚本。
> 核心能力：**判断力** — 知道何时用工具、何时写脚本、何时调API。

### 4.1 先搞清楚：工具 vs API

| 场景 | 用工具就够（Claude Code / Cursor） | 需要API脚本 |
|------|----------------------------------|-------------|
| 单次用例生成 | ✅ 直接对话 | ❌ 没必要 |
| 单次代码审查 | ✅ 直接对话 | ❌ 没必要 |
| 批量生成100个功能的用例 | ❌ 手动重复太多 | ✅ 脚本循环调用 |
| CI流水线自动分析 | ❌ 需要无人值守 | ✅ 脚本自动触发 |
| 定时质量报告 | ❌ 需要定时触发 | ✅ cron + 脚本 |

**判断标准**：手动做超过3次的、需要定时/自动触发的、需要批量处理的 → 用API脚本。其他用工具就够。

### 4.2 调通API（只需要一次）
- 注册Claude API Key
- 用AI帮你写一个最简单的API调用脚本
- 理解：代码发prompt → API返回结果 → 代码处理结果

### 4.3 值得用API自动化的场景

只自动化真正重复的操作：

| 场景 | 输入 | 输出 | 为什么值得 |
|------|------|------|-----------|
| 批量生成用例 | 需求文档 | 禅道可导入的CSV | 每个版本都要做 |
| 测试报告摘要 | pytest JSON报告 | 文字摘要 | 每天都跑 |
| Bug归因 | 失败日志 | 分类后的问题清单 | CI自动触发 |

### 4.4 输出质量自动校验
- 用例生成后，脚本自动检查：是否有空字段、格式是否正确
- 数据生成后，脚本自动验证：是否符合字段约束
- **不能盲信AI输出，代码要加校验逻辑**

### 4.5 CI/CD集成

**最小可行方案**：
```
代码提交 → CI触发 → 运行pytest → 测试结果JSON
    ↓
post-test脚本调用AI API → 失败原因分类 + 建议操作
    ↓
结果推送到企业微信/钉钉/邮件
```

**关键点**：CI环境中API调用要加超时和降级（AI API不可用时不阻塞构建）。

### 4.6 Prompt模板管理

```
prompts/
├── test_case_gen.txt      # 用例生成模板
├── data_gen.txt           # 数据生成模板
├── bug_analysis.txt       # 缺陷分析模板
├── report_summary.txt     # 报告摘要模板
└── security_scan.txt      # 安全测试模板

每个模板用变量占位符：{product_name}、{module}、{requirement}
脚本自动填充变量 → 调用API → 校验输出
```

- [ ] API调用跑通
- [ ] 至少2个真正重复场景的自动化脚本可用
- [ ] 脚本包含AI输出校验逻辑
- [ ] 一套prompt模板目录，至少5个模板

## 阶段五：AI Agent + MCP打通工作流（第9-11周）

### 目标：用Agent串联流程，用MCP连接禅道等外部工具

> 修订说明：MCP生态已成熟，从"选做"升级为推荐项。
> 连通禅道和CI是你日常提效最大的一步。

- Claude Code本身就是一个Agent — 它能读文件、写代码、执行命令

### 5.4 多Agent协作模式

**示例流程**：

## ★ 应对快速迭代：AI时代的质量防线

> **现实问题**：需求变更频繁、版本迭代快、开发也在用AI写代码和修bug——代码量暴增但质量不可控。
> 测试工程师面对的不再只是"测功能"，而是"在混沌中守住质量底线"。

### 问题拆解

| 挑战 | 根因 | 对测试的影响 |
|------|------|-------------|
| 需求频繁变更 | 市场/客户驱动，短周期迭代 | 用例反复失效，回归范围不断膨胀 |
| 开发用AI写代码 | AI代码看着能跑，但边界处理、安全考虑往往不足 | 表面通过率高，隐性缺陷增多 |
| 开发用AI修bug | AI倾向于"头痛医头"，patch式修复 | 引入新问题、改出回归bug |
| 版本发布节奏快 | 测试时间被压缩 | 没时间全量回归，只能选择性测试 |

### 应对策略一：变更影响分析自动化

**每次需求变更/代码提交，先问AI"这改了什么、影响什么"**

```
以下是本次版本的变更内容：
"""
[粘贴需求变更说明 或 git diff/commit log]
"""

产品背景：安全/网络/密码一体机，模块包括：[列出主要模块]

请分析：
1. 本次变更直接影响哪些模块？
2. 间接影响哪些模块？（依赖关系、共享组件、数据流关联）
3. 哪些已有测试用例需要更新？（标注用例编号或场景描述）
4. 哪些已有测试用例可能失效？（功能改了但用例没跟上）
5. 建议的回归测试范围（最小集 vs 安全集）
6. 风险评估：这次变更引入回归bug的可能性（高/中/低）+ 理由
```

**关键**：把这个分析嵌入每次提测流程，不需要额外花时间，10分钟跑完。

### 应对策略二：AI代码的专项质量检查

**开发用AI写的代码，你要知道AI代码的典型质量问题**

| AI代码常见问题 | 怎么测 | 提示词方向 |
|---------------|--------|-----------|
| 只处理了happy path | 用异常输入/边界值轰炸 | "生成该接口的异常测试用例，重点覆盖AI容易忽略的边界" |
| 错误处理是空壳（catch了但没处理） | 故意触发错误，看是否有正确响应 | "检查这段代码的错误处理是否完整" |
| 安全校验缺失 | 用安全扫描脚本过一遍 | 阶段二的2.5安全专项脚本 |
| 硬编码配置/密钥 | 静态扫描 | 阶段二的2.4静态分析 |
| 并发/竞态问题 | 多线程压测 | 阶段二的2.6性能测试 |
| 修一个bug引入另一个 | 变更影响域的回归测试 | 上面的策略一 |

**提示词模板（审查AI生成的修复代码）**：
```
开发使用AI修复了以下bug，请审查修复代码：

原始bug：[bug描述]
修复diff：
"""
[粘贴代码变更]
"""

请评估：
1. 修复是否真的解决了根本原因？还是只是表面修补？
2. 修复是否可能引入新问题？（副作用分析）
3. 修复范围是否完整？（同类问题在其他地方是否也存在？）
4. 需要针对这次修复补充哪些回归测试用例？
```

### 应对策略三：精准回归 — 时间不够时怎么选

```
测试时间有限（只有x天），以下是本版本的全部变更和bug修复列表：
"""
[变更列表]
"""

历史数据：
- 上版本回归中发现的bug分布：[模块→bug数]
- 高风险模块清单：[基于3.5的缺陷预测]

请帮我制定精准回归策略：
1. 必测项（P0，不测不能发版）
2. 应测项（P1，时间允许必须测）
3. 可选项（P2，风险较低可以跳过）
4. 每个项标注：预计耗时、跳过的风险
5. 如果只有1天/2天/3天，分别建议的测试范围
```

### 应对策略四：建立变更追踪机制

**实操方法**：
```
维护一个"变更-用例映射表"：

| 变更ID | 变更描述 | 影响模块 | 关联用例 | 用例状态 | 回归结果 |
|--------|---------|---------|---------|---------|---------|
| CHG-001 | 修改VPN配置接口参数 | VPN、网络策略 | TC-101~TC-115 | 已更新 | 通过 |

每次需求变更时，AI帮你：
1. 自动识别受影响的用例
2. 标注哪些用例需要更新
3. 生成更新后的用例
4. 跟踪回归测试状态
```

**好处**：需求再怎么变，你都能快速定位"哪些测试要跟着变"，不会遗漏。

## ★ 让AI理解你的业务：业务知识注入方法

> **核心矛盾**：AI什么都懂一点，但不懂你的产品。它知道"防火墙"的概念，但不知道你的一体机防火墙规则的具体约束、业务规则、用户操作习惯。
> 结果：生成的用例"看着像那么回事但经不起推敲"。

### 为什么AI的业务测试不准？

| 表现 | 原因 |
|------|------|
| 生成了你产品不存在的功能 | AI基于通用知识猜测，不了解你的产品范围 |
| 测试步骤在真机上操作不通 | AI不知道你的页面交互流程和实际UI布局 |
| 边界值不对（如端口范围、IP限制） | AI用通用标准，不知道你的产品有自定义限制 |
| 遗漏了业务组合场景 | AI不知道"配置VPN时必须先配好证书"这种业务依赖 |
| 安全测试脱离产品实际 | AI不知道你产品用了哪些安全机制、哪些已经防护了 |

### 方法一：构建产品知识摘要（最重要）

**一次性投入，长期复用。** 写一份你产品的"AI背景文档"：

```markdown
# [产品名]AI测试知识库

## 产品概述
- 产品类型：安全/网络/密码一体机
- 用户角色：系统管理员（超级管理员、审计管理员、普通管理员）
- 管理方式：Web管理页面（HTTPS，端口443）

## 功能模块清单
| 模块 | 功能 | 关键业务规则 |
|------|------|-------------|
| 用户管理 | 账号增删改查、角色分配 | 三权分立，管理员不能自己改自己的权限 |
| 网络配置 | IP/路由/VLAN/防火墙 | 防火墙规则最多500条，按序匹配 |
| VPN配置 | IPSec/SSL VPN | 必须先导入证书才能配置SSL VPN |
| 密码服务 | SM2/SM3/SM4加解密 | 符合GM/T 0054标准 |
| 日志审计 | 操作日志、安全日志 | 日志不可删除、不可篡改 |

## 字段约束（AI常犯错的点）
- IP地址：支持IPv4和IPv6，IPv4段不能跨网段
- 端口：1-65535，系统保留端口（1-1024）需要管理员权限
- 密码：8-32位，必须包含大小写+数字+特殊字符
- 会话超时：5-60分钟，默认30分钟
- 设备型号：[列出你的型号]，不同型号接口数不同

## 模块依赖关系
- 修改网络配置 → 影响VPN和防火墙
- 证书过期 → SSL VPN不可用
- 修改密码策略 → 已有用户下次登录强制改密

## 已知限制/特殊行为
- 同时只支持1个管理员登录操作（后登录的踢掉前一个）
- 防火墙规则修改后需要手动"应用"才生效
- 恢复出厂设置不会清除审计日志
```

**使用方法**：每次给AI的prompt开头加一句：
```
请先阅读以下产品知识库，所有测试都要基于这些约束：
"""
[粘贴知识库内容]
"""
```

### 方法二：场景链式注入（逐步喂信息）

**适用于复杂业务流程测试**：

```
第1轮：给AI业务流程图
"以下是我们VPN配置的完整流程：
 ① 导入CA证书 → ② 创建VPN实例 → ③ 配置加密参数 → ④ 绑定接口 → ⑤ 添加用户 → ⑥ 启用VPN
 每一步失败都应该有明确的错误提示，不能静默失败。"

第2轮：给AI一个真实操作的截图或录屏描述
"实际操作时，步骤③的界面长这样：[描述或截图]，
 可选的加密算法是SM1/SM2/SM4，不是通用的AES/DES。"

第3轮：给AI一个真实bug案例
"我们之前遇到过的bug：在步骤④绑定接口时，如果选了已被防火墙占用的接口，
 系统没报错但VPN无法建立连接。请基于这种'静默失败'的模式，找其他步骤中可能也有类似问题的地方。"
```

**核心思路**：不要指望一次prompt就让AI理解你的业务。像带新人一样，分步骤教。

### 方法三：用真实数据校准AI

```
我用AI生成了以下测试用例：
"""
[粘贴AI生成的用例]
"""

以下是我在真实设备上执行的结果：
- 用例3：步骤2操作不了，实际界面没有这个按钮
- 用例7：预期结果错误，实际上系统允许空密码（这是个bug还是用例错了？）
- 用例12：这个场景在我们产品上不存在

请基于这些反馈：
1. 修正以上有问题的用例
2. 分析为什么会生成这些错误（是缺了什么背景信息？）
3. 告诉我下次prompt中应该补充什么信息来避免这些问题
```

**关键**：每次AI犯错都是优化prompt的机会。3-5轮后，AI的输出准确率会显著提升。

### 方法四：建立"AI易错点"清单（持续积累）

```markdown
# AI业务测试易错点（持续更新）

## 用户管理模块
- ❌ AI总是假设"管理员可以改自己的密码"→ 实际上需要通过审计管理员
- ❌ AI不知道三权分立，生成的权限用例缺少角色交叉测试
- ✅ 补充prompt后解决："我们的产品实行三权分立..."

## 网络配置模块
- ❌ AI生成的IP地址用了保留段（如127.x.x.x、169.254.x.x）
- ❌ AI不知道我们的子网掩码只支持到/30
- ✅ 在知识库字段约束中补充后解决

## VPN模块
- ❌ AI不知道"必须先导入证书"这个前置条件
- ❌ AI用AES/DES，不知道我们只支持国密算法
- ✅ 在模块依赖关系中补充后解决
```

**这个清单价值极高**：它让你的prompt越来越精准，也可以分享给团队其他人用。

### 准确度提升预期

| 阶段 | AI业务测试准确度 | 关键动作 |
|------|-----------------|---------|
| 裸用（不加背景） | ~30-40% 可用 | 很多用例需要大改 |
| 加产品知识摘要 | ~60-70% 可用 | 大方向对了，细节仍需修正 |
| 加字段约束+模块依赖 | ~75-85% 可用 | 少数需要微调 |
| 加易错点清单+真实反馈 | ~85-90% 可用 | 基本可直接用，少量审查 |

**永远到不了100%**——这正是测试工程师的价值所在。AI做80%的体力活，你把关最后20%的判断。

## ★ 质量门禁 — 每个阶段的通过/不通过标准

**为什么需要**：没有门禁的流程只是走过场。AI可以帮你定义和执行质量门禁。

### 各阶段质量门禁

| 阶段 | 质量门 | AI怎么帮 | 不通过怎么办 |
|------|--------|----------|-------------|
| 需求评审 | 需求文档无模糊项、无遗漏场景、可测试 | AI预审找问题 | 打回给产品，列出具体问题 |
| 用例评审 | 覆盖率达标（正常/异常/边界/安全全覆盖） | AI评估覆盖度 | 补充遗漏维度的用例 |
| 提测准入 | 冒烟测试100%通过、关键接口可用 | AI生成冒烟用例 | 打回开发，附冒烟失败详情 |
| 测试退出 | P0/P1 bug为0、P2 bug≤3、回归通过率≥95% | AI统计并判断 | 列出未达标项，评估风险 |
| 发布验证 | 生产冒烟通过、关键监控指标正常 | AI对比基线 | 回滚并启动故障排查 |

**提示词模板（退出标准判断）**：
```
以下是本轮测试的汇总数据：
"""
总用例数：xx | 通过：xx | 失败：xx | 阻塞：xx
遗留bug：P0=x, P1=x, P2=x, P3=x
回归测试通过率：xx%
"""

退出标准：P0/P1=0，P2≤3，回归通过率≥95%

请判断：
1. 是否满足发版条件？逐项对比
2. 如果不满足，差距在哪？
3. 遗留bug的风险评估（哪些可以带病上线，哪些必须修）
4. 给出你的发版建议（发/不发/条件发布）
```

## ★ 持续改进闭环（PDCA）

**测试不是一次性的，全质量管理的核心是持续改进。**

### 每个版本的改进循环

```
Plan（计划）→ 版本测试策略（AI辅助制定，基于上版本的数据）
  ↓
Do（执行）→ 按策略执行测试（AI辅助生成用例/脚本/数据）
  ↓
Check（检查）→ 版本质量复盘（AI分析本版本数据 vs 上版本）
  ↓
Act（改进）→ 更新测试策略/用例库/自动化覆盖（AI建议改进项）
  ↓
→ 反哺下一轮Plan
```

### 版本质量复盘（每版本必做）

**提示词模板**：
```
以下是本版本的质量数据：
"""
版本号：[xx]
测试周期：[起止日期]
bug总数：xx（P0=x, P1=x, P2=x, P3=x）
bug分布：[按模块统计]
线上逃逸bug：x个（描述）
测试用例总数/新增/通过率：xx/xx/xx%
自动化覆盖率：xx%
"""

上版本数据：[同样的指标]

请输出版本质量复盘报告：
1. 质量趋势对比（本版本 vs 上版本，哪些指标改善/恶化）
2. 本版本最大质量风险是什么？
3. 线上逃逸bug分析（为什么没在测试阶段发现？）
4. 测试过程中哪个环节效率最低？
5. 下版本的3条具体改进建议（可执行的，不要泛泛而谈）
```

### 测试资产持续积累

| 资产 | 怎么积累 | AI怎么帮 |
|------|---------|----------|
| 用例库 | 每版本AI生成 + 人工审查后入库 | AI去重、标注覆盖维度 |
| 自动化脚本 | 从高频回归场景开始自动化 | AI写脚本、维护失败脚本 |
| Prompt模板库 | 每次迭代后保留效果好的prompt | AI帮你对比不同prompt的效果 |
| 缺陷模式库 | 记录产品的典型缺陷模式 | AI从历史bug中提取模式 |
| 检查清单 | 需求评审/代码评审/测试退出清单 | AI基于历史问题持续补充 |

## ★ AI效果度量 — 怎么知道AI真的有用？

**不度量就不知道值不值得继续投入。** 从阶段一开始记录，不需要精确，但要有数据。

### 效率指标

| 指标 | 怎么量 | 记录方式 |
|------|--------|----------|
| 用例设计时间 | 同一功能：手工 vs AI辅助各花多久 | 计时对比 |
| 用例覆盖度 | AI生成的用例 vs 你手写的，互相查漏 | 统计遗漏数 |
| 脚本开发时间 | 让AI写 vs 自己写/改别人的 | 计时对比 |
| Bug发现率 | AI辅助测试周期 vs 之前，发现的bug数 | 按版本统计 |

### 质量指标

| 指标 | 怎么量 | 说明 |
|------|--------|------|
| AI输出可用率 | 生成10条用例，几条不改就能用？ | 越高说明prompt越好 |
| AI遗漏率 | AI生成后你补了几条？ | 越低说明覆盖越全 |
| AI幻觉率 | 生成的内容里有几条是编造的？ | 越低说明约束越好 |
| 回归bug减少量 | 自动化覆盖后，回归bug是否减少 | 按版本对比 |

### 简单记录模板

```
日期：2026-xx-xx
场景：[用AI做了什么]
耗时：AI辅助 xx 分钟 / 估计手工 xx 分钟
AI输出质量：可直接用 x 条 / 需修改 x 条 / 不可用 x 条
发现的问题：[AI遗漏了什么 / 编造了什么]
改进：[下次prompt怎么调整]
```

**每周花10分钟回顾这些数据**，它会告诉你：哪些场景AI真的提效了，哪些场景不如手工。

## ★ 每个阶段的防坑指南

| 阶段 | 常见坑 | 怎么防 |
|------|--------|--------|
| 一 | AI需求评审发现的"问题"其实不是问题 | AI不了解业务上下文，你要判断真伪，只带真问题去评审 |
| 一 | AI生成的用例看着多但覆盖不全 | 用审查checklist逐项检查 |
| 一 | AI不了解你的产品，生成无关内容 | 提示词里加产品背景和约束 |
| 一 | 探索性测试建议太泛，不可操作 | 给AI更具体的模块信息和已知约束 |
| 二 | AI写的代码跑不通 | 把报错原样发给AI，让它修 |
| 二 | AI写的代码逻辑对但断言错 | 你来审查断言是否符合预期 |
| 二 | 安全测试payload被产品WAF拦截 | 区分"被正确拦截"和"测试本身有问题" |
| 二 | 性能测试脚本并发数设太高打崩环境 | 从小并发开始，逐步加压 |
| 三 | 禅道格式不匹配 | 先导入1条试，确认格式再批量 |
| 三 | 缺陷预测结果和直觉不符 | 检查输入数据质量，预测仅供参考不替代判断 |
| 三 | 生产日志太多AI处理不了 | 先筛选关键日志（ERROR/WARN级别），分批给AI |
| 三 | 故障复盘变成甩锅分析 | 引导AI聚焦"流程改进"而非"谁的责任" |
| 四 | API返回结果不稳定 | 加重试机制 + 结果校验 |
| 四 | CI中AI API调用超时阻塞流水线 | 设超时 + 降级策略（AI不可用时跳过分析） |
| 五 | 想一步到位搭复杂系统 | 先跑通最简流程，再逐步加功能 |
| 通用 | 需求频繁变更，用例跟不上 | 建立变更-用例映射表，AI自动识别受影响的用例 |
| 通用 | 开发用AI修的bug引入新问题 | 每次修复都做变更影响分析+定向回归 |
| 通用 | AI不懂业务，生成的用例不准 | 构建产品知识摘要，逐步积累AI易错点清单 |
| 通用 | 知识摘要写了但AI还是犯错 | 用真实执行结果反馈给AI校准，3-5轮后显著改善 |

**通用原则：AI是工具，你是质量把关人。AI生成的任何东西，上线/提交前都要经过你的审查。**

### ★ 月度AI校准（每月1小时）

**目的**：不追新闻，只校准"我的工作有没有更好的做法"。

**每月做一次**：
1. **试新功能（30分钟）**：你正在用的AI工具（Claude Code等）更新了什么？挑一个新功能，用你手头的测试任务试一次。有用就留，没用就跳过。
2. **回顾效果（20分钟）**：翻看本月的AI效果记录表，哪些场景AI帮了大忙？哪些场景还是手工更快？
3. **调整一件事（10分钟）**：基于回顾，只改一个具体做法。比如"下个月用例生成换成新的prompt模板"或"某个手工步骤试试用MCP自动化"。

**判断标准**：看到AI新闻时问自己 — "这能帮我更快地测完一体机的某个功能吗？" 能→花30分钟试试。不确定→先收藏。不能→跳过。
