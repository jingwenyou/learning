《性能之巅》性能分析方法论解读

一、核心方法论
================================================================================

1. USE方法论 (Utilization-Saturation-Errors)
--------------------------------------------------------------------------------
作者: Brendan Gregg
适用场景: 所有资源类型

┌─────────────────────────────────────────────────────────────────────┐
│                        USE方法论                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Utilization (利用率)  ← 资源忙碌时间百分比                         │
│          ↓                                                          │
│   Saturation (饱和度)  ← 资源排队/等待工作量                        │
│          ↓                                                          │
│   Errors (错误率)     ← 错误发生频率                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

解读:

| 维度       | 定义                 | 采集方法                         | 阈值示例     |
|------------|---------------------|--------------------------------|-------------|
| Utilization | 资源忙碌时间%       | psutil.cpu_percent()           | CPU>80%告警 |
| Saturation | 排队工作量          | load_average, swap_percent     | 负载>2告警  |
| Errors     | 错误频率            | errin, errout, dropin, dropout | 重传>1%告警 |

为什么先查Saturation:
"If there is saturation, do not immediately look at Utilization" — Brendan Gregg

饱和度是瓶颈的更直接信号，利用率高不一定是问题（可能只是资源充足）。


2. RED方法 (Rate-Errors-Duration)
--------------------------------------------------------------------------------
适用场景: 微服务/API监控

┌─────────────────────────────────────────────────────────────────────┐
│                        RED方法                                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Rate (速率)        ← 每秒请求数 (QPS/RPS)                         │
│          ↓                                                          │
│   Errors (错误率)   ← 错误请求百分比                                │
│          ↓                                                          │
│   Duration (延迟)   ← 请求响应时间 (P50/P90/P99)                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

与USE对比:

| 方法 | 适用场景      | 核心指标           |
|------|--------------|-------------------|
| USE  | 系统资源      | CPU/内存/磁盘/网络 |
| RED  | 服务/应用      | QPS/错误率/延迟   |

二、关键系统性能指标
================================================================================

1. CPU指标体系
--------------------------------------------------------------------------------

┌─────────────────────────────────────────────────────────────────────┐
│                     CPU 利用率分解                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  100% ──────────────────────────────────                          │
│       │ user │ nice │ system │ iowait │ irq │ softirq │          │
│       │  15% │  0%  │   5%  │   2%   │ 0%  │   1%    │          │
│       └───────────────────────────────────                          │
│                                                                     │
│  《性能之巅》重点关注:                                               │
│  • iowait > 20%: 存在I/O瓶颈，需检查磁盘                          │
│  • irq + softirq > 10%: 中断处理过多，需优化                       │
│  • steal > 1%: 虚拟化环境CPU竞争                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

| 指标     | 读取源                     | 含义             | 阈值              |
|---------|---------------------------|------------------|------------------|
| user    | cpu_times_percent.user    | 用户态CPU        | >80%可能需优化   |
| system  | cpu_times_percent.system  | 内核态CPU        | >30%异常         |
| iowait  | cpu_times_percent.iowait  | CPU等待I/O       | >20%磁盘瓶颈     |
| irq     | cpu_times_percent.irq     | 硬中断处理       | >10%过多         |
| softirq | cpu_times_percent.softirq | 软中断处理       | >10%过多         |
| steal   | steal                    | 虚拟化抢占       | >1%资源竞争      |

CPU饱和度指标:

| 指标         | 含义             | 阈值         |
|-------------|-----------------|--------------|
| load_average | 系统负载(1/5/15min) | >CPU核心数告警 |
| run_queue   | 运行队列长度     | >4告警        |
| procs_running | 可运行进程数  | 持续>4异常    |


2. 内存指标体系
--------------------------------------------------------------------------------

┌─────────────────────────────────────────────────────────────────────┐
│                     内存使用分解                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  total: 16GB                                                      │
│  ├── available: 6GB (包括buffers/cached)                          │
│  │   ├── buffers: 0.5GB                                          │
│  │   ├── cached: 4GB                                             │
│  │   └── free: 1.5GB                                             │
│  └── used: 10GB (实际应用程序使用)                                 │
│                                                                     │
│  《性能之巅》重点关注:                                              │
│  • available ≈ free + buffers + cached                            │
│  • swap > 0%: 内存已饱和，启用交换                                 │
│  • iowait高但CPU不高: 可能是内存回收导致                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

| 指标     | 读取源              | 含义       | 阈值        |
|---------|-------------------|-----------|------------|
| total   | virtual_memory.total | 总内存    | -          |
| available | virtual_memory.available | 可用内存 | <20%告警   |
| percent | virtual_memory.percent | 使用率    | >80%告警   |
| swap    | swap_memory.percent | 交换使用率 | >10%饱和信号 |
| buffers | virtual_memory.buffers | 缓冲区   | -          |
| cached  | virtual_memory.cached  | 缓存      | -          |


3. 磁盘I/O指标体系
--------------------------------------------------------------------------------

┌─────────────────────────────────────────────────────────────────────┐
│                     磁盘I/O 分析                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  iostat 关键指标:                                                   │
│                                                                     │
│  r/s, w/s      ← 每秒读写次数 (IOPS)                              │
│  rkB/s, wkB/s  ← 每秒读写速度 (吞吐量)                            │
│  await          ← 平均I/O响应时间 (含排队)                         │
│  %util          ← 设备利用率 (接近100%=饱和)                       │
│  avgqu-sz       ← 平均请求队列长度                                  │
│                                                                     │
│  《性能之巅》性能分析:                                               │
│                                                                     │
│  场景1: await > 10ms, %util > 80%                                  │
│    → 磁盘I/O饱和，需要:                                            │
│      • 升级到SSD                                                   │
│      • 增加磁盘并行度                                               │
│      • 优化I/O模式(顺序/随机)                                      │
│                                                                     │
│  场景2: await高但%util低                                            │
│    → 可能是单个大I/O请求延迟高                                      │
│      • 检查硬件故障                                                 │
│      • 检查文件系统碎片                                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

| 指标       | 含义             | 正常  | 告警  | 危险  |
|-----------|-----------------|-------|-------|-------|
| r/s       | 读IOPS          | -     | -     | -     |
| w/s       | 写IOPS          | -     | -     | -     |
| await     | 平均响应时间     | <5ms  | >10ms | >50ms |
| %util     | 设备利用率       | <60%  | >80%  | >90%  |
| avgqu-sz  | 平均队列长度     | <2    | >4    | >16   |


4. 网络指标体系
--------------------------------------------------------------------------------

┌─────────────────────────────────────────────────────────────────────┐
│                     网络性能分析                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  吞吐量:                                                           │
│  • bytes_sent/recv — 总流量                                       │
│  • packets_sent/recv — 数据包数                                    │
│                                                                     │
│  错误率 (Errors):                                                  │
│  • errin, errout — 接收/发送错误                                   │
│  • dropin, dropout — 接收/发送丢包                                 │
│                                                                     │
│  延迟:                                                            │
│  • RTT (Round Trip Time)                                           │
│  • 可用 ping, traceroute 测量                                      │
│                                                                     │
│  《性能之巅》网络检查:                                               │
│                                                                     │
│  1. ifutil > 80%: 带宽饱和                                        │
│     → 考虑: 压缩、CDN、限流                                        │
│                                                                     │
│  2. 错误/丢包 > 0:                                                 │
│     → 检查: 网线、物理连接、驱动                                     │
│                                                                     │
│  3. 重传率 > 1%:                                                   │
│     → 网络不稳定或拥塞                                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

| 指标        | 含义       | 阈值      |
|------------|-----------|----------|
| bytes_sent/recv | 总流量  | -        |
| errin/errout  | 错误数     | >0异常   |
| dropin/dropout | 丢包数   | >0异常   |
| retransmits  | 重传率      | >1%异常  |

三、性能瓶颈定位思路
================================================================================

1. 排查流程图
--------------------------------------------------------------------------------

┌─────────────────┐
│  Step 1: 确定问题  │
│  用户反馈: 慢/卡/无响应  │
└────────┬────────┘
         ↓
┌─────────────────┐
│  Step 2: 系统概况   │
│  dstat -cdny 1  │
│  CPU/磁盘/网络/内存 │
└────────┬────────┘
         ↓
┌─────────────────┐
│  Step 3: 定位瓶颈  │
│  先检查Saturation  │
└────────┬────────┘
         ↓
    ┌────┴────┐
    ↓         ↓
 CPU高?    内存高?
    ↓         ↓
 iowait高?  swap使用?
    ↓         ↓
    ↓      磁盘I/O高?
    ↓         ↓
┌─────────────────┐
│  Step 4: 热点追踪  │
│  perf top      │
│  火焰图分析      │
└────────┬────────┘
         ↓
┌─────────────────┐
│  Step 5: 验证修复  │
│  重复Step 2-4   │
└─────────────────┘


2. 快速定位检查清单
--------------------------------------------------------------------------------

| 症状     | 首先检查     | 工具             |
|---------|------------|-----------------|
| CPU高    | iowait, load | top, mpstat     |
| 内存高    | swap, available | free -m, vmstat |
| 磁盘I/O高 | await, %util | iostat -xz 1   |
| 网络延迟  | 带宽, 错误率  | sar -n DEV 1    |
| 进程卡死  | 状态, 僵尸    | ps aux          |


3. 常见瓶颈与解决
--------------------------------------------------------------------------------

| 瓶颈类型   | 诊断信号            | 解决方案                    |
|-----------|-------------------|---------------------------|
| CPU瓶颈   | user>80%, load>4   | 优化算法, 多线程, 水平扩展  |
| 内存瓶颈   | available低, swap>0 | 释放缓存, OOM排查, 增加内存 |
| 磁盘I/O瓶颈 | await>10ms, %util>80% | SSD, RAID, 索引优化    |
| 网络瓶颈   | ifutil>80%, 重传>1%  | 压缩, CDN, 限流          |
