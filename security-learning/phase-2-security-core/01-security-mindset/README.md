# 01 安全思维 —— 从测试思维到攻击者思维

> 你已经有测试工程师的思维：**找到系统不按预期工作的地方**。
> 安全测试只是把这个思维延伸一步：**找到系统可以被恶意利用的地方**。
> 这个模块不讲具体漏洞，讲的是**怎么想**。

---

## 一、测试思维 vs 安全思维

### 你已经拥有的（测试思维）

```
功能测试这样想：
  "输入正确的用户名和密码，能不能登录成功？"
  "输入错误的密码，会不会提示错误？"
  "密码为空，会不会报错？"
  → 关注：系统是否按照需求工作
```

### 你需要加上的（安全思维）

```
安全测试这样想：
  "输入 admin' OR 1=1 -- 作为用户名，会发生什么？"
  "登录失败的提示是'用户名不存在'还是'密码错误'？这有什么区别？"
  "密码传输是明文还是加密的？中间能不能截获？"
  "登录成功后的Token能不能被预测？能不能被重放？"
  "用我的Token去访问别人的数据，服务端会检查吗？"
  → 关注：系统能不能被恶意利用
```

**核心区别：**

| 测试思维 | 安全思维 |
|---------|---------|
| 验证功能是否正确 | 寻找功能是否可被滥用 |
| 按照预设路径测试 | 故意偏离预设路径 |
| 输入合理的数据 | 输入恶意的数据 |
| 关注"能不能用" | 关注"能不能被利用" |
| 站在用户角度 | 站在攻击者角度 |

**好消息：** 两种思维不矛盾，安全思维是测试思维的超集。你已经走了一半路。

## 二、攻击者思维模型：CIA三角

安全的核心目标是保护三样东西：

```
         Confidentiality
          （机密性）
           /      \
          /        \
         /   CIA    \
        /            \
  Integrity ──── Availability
  （完整性）      （可用性）
```

**机密性（Confidentiality）** —— 信息不被未授权的人看到
```
攻击者的目标：偷看数据
漏洞举例：
  - SQL注入读取数据库 → 泄露用户密码
  - 越权访问别人的订单 → 泄露他人隐私
  - 目录遍历读取 /etc/passwd → 泄露系统信息
  - HTTPS降级为HTTP → 中间人偷看数据

你的测试目标：
  "我能看到不该看的数据吗？"
```

**完整性（Integrity）** —— 信息不被未授权的人修改
```
攻击者的目标：篡改数据
漏洞举例：
  - SQL注入修改数据 → 改价格、改权限
  - CSRF让用户在不知情时执行操作 → 转账、改密码
  - 越权修改别人的信息
  - XSS在页面注入虚假信息

你的测试目标：
  "我能改掉不该改的数据吗？"
```

**可用性（Availability）** —— 系统保持正常运行
```
攻击者的目标：让系统崩溃
漏洞举例：
  - DDoS攻击 → 请求量打满服务器
  - ReDoS → 用恶意正则让CPU 100%
  - 资源耗尽 → 上传超大文件占满磁盘

你的测试目标：
  "我能让系统不可用吗？"
  （注意：DoS测试要格外小心，容易影响真实服务）
```

## 三、STRIDE威胁模型：系统化地想

当你拿到一个功能要做安全测试时，用STRIDE模型逐条过一遍：

| 威胁 | 英文 | 含义 | 想什么 |
|------|------|------|--------|
| 仿冒 | Spoofing | 冒充他人身份 | "能不能伪造身份？Token能不能篡改？" |
| 篡改 | Tampering | 修改数据 | "请求参数、Cookie、本地存储能不能改？" |
| 抵赖 | Repudiation | 否认做过的事 | "操作有没有日志？日志能不能被删？" |
| 信息泄露 | Info Disclosure | 泄露敏感数据 | "错误信息、响应、源码有没有泄露内部信息？" |
| 拒绝服务 | Denial of Service | 让服务不可用 | "有没有限流？大量请求会怎样？" |
| 权限提升 | Elevation of Privilege | 获取更高权限 | "普通用户能做管理员的事吗？" |

### 实战举例：分析一个"修改个人信息"功能

```
功能描述：用户可以修改自己的昵称和头像

用STRIDE分析：

[S] 仿冒：
  - 不登录能调这个接口吗？
  - 用别人的Token能改别人的信息吗？

[T] 篡改：
  - 请求中有没有user_id参数？能不能改成别人的ID？
  - 除了昵称和头像，能不能顺便改role字段？
  - 头像上传能不能传一个webshell？

[R] 抵赖：
  - 修改操作有没有审计日志？

[I] 信息泄露：
  - 修改接口的响应有没有返回不该返回的字段（密码、手机号）？
  - 错误响应有没有暴露内部信息？

[D] 拒绝服务：
  - 头像上传有没有大小限制？传100MB的文件会怎样？
  - 修改请求有没有频率限制？

[E] 权限提升：
  - 普通用户能调管理员的修改接口吗？
  - 请求中加一个 "role": "admin" 字段会怎样？
```

## 四、攻击面分析：从哪里入手

### 4.1 什么是攻击面

攻击面 = **所有可以接收外部输入的地方**。

```
一个典型Web应用的攻击面：

[用户可控的输入]
├── URL路径和参数     /search?q=__这里__
├── POST请求体        {"username": "__这里__"}
├── HTTP请求头        Cookie: __这里__; User-Agent: __这里__
├── 文件上传          文件名、文件内容、Content-Type
├── URL片段           #__这里__（前端路由）
└── WebSocket消息     实时通信的数据

[容易被忽略的输入]
├── HTTP方法          GET改成DELETE会怎样？
├── Content-Type      换个格式服务端还接受吗？
├── Referer/Origin    CSRF防护依赖这些头？
├── Accept-Language   有些系统根据语言返回不同内容
└── X-Forwarded-For   伪造IP地址

[更隐蔽的入口]
├── 第三方集成        OAuth回调、Webhook、SSO
├── API文档           Swagger/OpenAPI暴露所有接口
├── 错误页面          404/500的详细错误信息
├── robots.txt        暴露不希望被访问的路径
└── 配置文件泄露      .git/, .env, backup文件
```

### 4.2 信息收集：测试前必做的事

正式测试之前，先了解目标：

```
第1步：被动信息收集（不直接接触目标）
  - 查看网站的公开页面、帮助文档
  - 查看robots.txt → 哪些路径不希望被访问
  - 查看HTML源码中的注释和JS文件
  - 查看HTTP响应头 → Server头暴露什么技术栈

第2步：主动信息收集（与目标交互）
  - 遍历所有页面和功能，记录所有接口
  - 用F12观察所有网络请求
  - 识别技术栈：后端语言、框架、数据库、Web服务器
  - 识别认证方式：Cookie/JWT/OAuth

第3步：绘制攻击面地图
  记录下来：
  - 所有API接口及其参数
  - 所有用户输入点
  - 所有文件上传功能
  - 所有需要认证/授权的功能
  - 认证和会话管理机制
```

## 五、OWASP Top 10：安全测试的"考试大纲"

OWASP（开放Web应用安全项目）总结了最常见、最危险的10类Web安全风险。
**这就是你接下来要逐一深入学习的内容。**

### OWASP Top 10 (2021版) 总览

```
A01: 访问控制失效 (Broken Access Control)
     → 越权访问，IDOR
     → "我能做/看不该做/看的事吗？"

A02: 加密机制失效 (Cryptographic Failures)
     → 明文传输密码，弱哈希，敏感数据泄露
     → "敏感数据保护得够不够？"

A03: 注入 (Injection)
     → SQL注入，命令注入，XSS
     → "我的输入会被当成代码执行吗？"

A04: 不安全设计 (Insecure Design)
     → 业务逻辑漏洞，缺少安全设计
     → "这个功能在设计上就有问题吗？"

A05: 安全配置错误 (Security Misconfiguration)
     → 默认密码，不必要的功能开启，详细错误信息
     → "配置是安全的吗？"

A06: 自带缺陷和过时的组件 (Vulnerable Components)
     → 使用了有已知漏洞的库/框架
     → "依赖有没有已知漏洞？"

A07: 身份识别和认证失效 (Auth Failures)
     → 暴力破解，弱密码，会话管理缺陷
     → "认证机制可靠吗？"

A08: 软件和数据完整性失效 (Integrity Failures)
     → 不安全的反序列化，CI/CD安全
     → "数据和代码的完整性有保证吗？"

A09: 安全日志和监控失效 (Logging Failures)
     → 缺少日志，无法检测攻击
     → "攻击发生后能不能被发现？"

A10: 服务端请求伪造 (SSRF)
     → 让服务器访问内部资源
     → "服务器会帮我访问内网吗？"
```

## 六、安全测试流程：你的工作方式

```
[1] 了解目标
    ├── 功能是什么？
    ├── 技术栈是什么？
    └── 谁可以用？有哪些角色？

[2] 信息收集
    ├── 遍历功能，记录所有接口
    ├── 分析认证/授权机制
    └── 记录所有输入点

[3] 威胁分析
    ├── 用STRIDE过一遍每个功能
    ├── 用OWASP Top 10过一遍整个应用
    └── 列出测试清单

[4] 执行测试
    ├── 从高风险功能开始（登录、支付、权限）
    ├── 每个输入点都试一遍常见payload
    └── 记录所有异常行为

[5] 验证和报告
    ├── 确认发现的问题确实可以被利用
    ├── 评估风险等级
    └── 写清复现步骤
```

## 七、动手实践

### 实践1：分析一个真实应用的攻击面

```
选择一个你常用的Web应用（或者用靶场），完成以下任务：

1. 列出所有主要功能（注册、登录、查看、修改、搜索……）
2. 对每个功能：
   - 记录HTTP请求的方法和URL
   - 记录所有参数
   - 标记哪些参数是用户可控的
3. 对登录功能用STRIDE模型做一次完整分析
4. 列出你认为风险最高的3个功能点，说明原因
```

### 实践2：信息收集实战

```bash
# 对DVWA靶场做一次信息收集

# 1. 查看响应头，识别技术栈
curl -I http://localhost:8081

# 2. 查看robots.txt
curl http://localhost:8081/robots.txt

# 3. 查看页面源码，找注释和隐藏信息
curl -s http://localhost:8081 | grep -i "<!--"
curl -s http://localhost:8081 | grep -iE "(password|secret|key|token|api|admin|todo|fixme|debug)"

# 4. 用F12遍历每个页面，记录所有请求
#    制作一个表格：功能 | URL | 方法 | 参数

# 5. 查看Cookie属性
#    F12 → Application → Cookies
#    检查 HttpOnly, Secure, SameSite
```

### 实践3：建立你的第一个安全测试清单

```
对DVWA的"登录功能"写一份安全测试清单：

认证测试：
□ 默认密码能不能登录（admin/admin, admin/password）
□ 连续登录失败有没有锁定/限频
□ 错误提示是"用户不存在"还是"密码错误"（用户名枚举）
□ 登出后Session是否失效
□ Session ID是否随机（不可预测）

输入测试：
□ 用户名输入SQL注入payload: admin' OR 1=1 --
□ 用户名输入XSS payload: <script>alert(1)</script>
□ 密码字段有没有长度限制
□ 特殊字符是否被正确处理

传输安全：
□ 是HTTP还是HTTPS
□ 密码在请求中是明文还是加密

会话管理：
□ Cookie有没有HttpOnly
□ Cookie有没有Secure
□ Session过期时间是否合理
```
