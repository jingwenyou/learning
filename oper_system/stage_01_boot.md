# 阶段 1：计算机启动

## 上节课回顾

无，这是第一课。

---

## 1. 按下电源，发生了什么？

**大白话：**

你按了一下电脑的电源按钮。电脑里有一块芯片，叫 **BIOS**（现在叫 UEFI），它住在主板上一个小小的 ROM 里，断电也不会丢数据。

> **ROM 是什么？** ROM = Read-Only Memory（只读存储器），和内存（RAM）不同，断电后数据不会丢失。BIOS 程序就烧录在 ROM 里，所以电脑刚通电就能找到它。

BIOS 干的第一件事：**自检**。

- 内存有多大？
- 键盘鼠标有没有？
- 启动顺序是什么？（先看硬盘，还是先看 U 盘？）

这大概花几秒钟。你看到"按 DEL 进入 BIOS"就是在这个阶段。

---

## 2. 为什么是 BIOS？这是历史包袱

**设计决策：**

1970 年代的计算机没有硬盘，都是打孔卡或者磁带。计算机需要一个"刚上电时第一个运行的程序"来加载真正的操作系统。

这个"第一个运行的程序"就是 BIOS——**Basic Input/Output System**。

| 选择 | 为什么 | 代价 |
|------|--------|------|
| 把启动程序焊在主板上 | 不依赖任何外部设备 | 更新困难 |
| 用磁盘/硬盘作为启动介质 | 灵活，可以换系统 | 需要一个标准来定义"怎么从磁盘加载" |

IBM PC 在 1981 年定了 BIOS 标准，兼容性一直留到现在。哪怕你的电脑是 2024 年的，BIOS 仍然存在。

**历史渊源：**

UEFI 是 BIOS 的"现代化版本"，解决了 BIOS 的一些限制（比如磁盘不能超过 2TB），但基本思路是一样的。

---

## 3. Bootloader：把内核从磁盘搬到内存

**大白话：**

BIOS 自检完了，下一步是找个能启动的东西。它会按启动顺序挨个问：

- "这个硬盘能启动吗？"
- "这个 U 盘能启动吗？"

找到第一个能启动的之后，加载 Bootloader，然后跳过去执行。

**传统 BIOS（MBR 模式）和现代 UEFI 的区别：**

| | 传统 BIOS + MBR | 现代 UEFI + GPT |
|---|---|---|
| 启动方式 | 加载磁盘最开头的 **512 字节**（MBR），然后执行 | 直接从 **EFI 分区** 加载 `.efi` 文件（可以很大） |
| 512 字节限制 | 有，所以需要分阶段加载 | 没有，直接加载完整的 Bootloader |
| 磁盘大小限制 | 最大 2TB | 无实际限制 |
| 现在的状态 | 老电脑还在用 | 2012 年之后的电脑基本都是 UEFI |

**传统 BIOS 为什么是 512 字节？** 这是历史决定的。1981 年的 IBM PC 软盘是主要启动介质，软盘的第一个扇区正好是 512 字节。就这么定下来了。512 字节放不了多少代码，所以传统 Bootloader 的任务是：**再把自己"变大"，去加载真正的操作系统内核**。

UEFI 解决了这个问题——它能直接读文件系统，所以不受 512 字节限制。

**常见的 Bootloader：**

- **GRUB**（Linux 主流，BIOS 和 UEFI 都支持）
- **systemd-boot**（简单，轻量，仅 UEFI）
- **Windows Boot Manager**（仅 UEFI）

---

## 4. 内核：操作系统的核心

**大白话：**

内核是操作系统最核心的部分。它负责：

- 管理 CPU（让多个程序"同时"运行）
- 管理内存（给每个程序分配独立的内存空间）
- 管理外设（键盘、鼠标、网卡、硬盘……）
- 提供系统调用（让应用程序能请内核帮忙）

内核加载到内存之后，CPU 就开始执行内核的代码了。

**启动过程中的一个关键转折：实模式 → 保护模式**

CPU 刚通电时运行在"实模式"（Real Mode），这是 1978 年 8086 CPU 的模式，只能访问 1MB 内存，没有内存保护。内核启动的早期阶段会切换到"保护模式"（Protected Mode），之后才能访问全部内存、实现进程隔离。

> 你不需要记住细节，只需要知道：**CPU 从"古老模式"切换到"现代模式"是启动过程中必须发生的一步**。

**典型数值：**

| 操作 | 典型耗时 |
|------|---------|
| BIOS 自检 | 1-5 秒 |
| 内核加载（Linux） | 1-3 秒 |
| 内核初始化完成 | 1-2 秒 |
| 第一个用户进程启动 | 0.1-0.5 秒 |

---

## 5. init：第一个用户进程

**大白话：**

内核初始化完了，不是直接给你看桌面。它会运行**第一个用户进程**，这个进程叫 **init**（在 Linux 里现在通常是 systemd）。

init 的任务是：启动更多进程，最终把登录界面跑起来。

```
启动序列（简化版）：

电源
  → BIOS/UEFI 自检
    → Bootloader（加载内核）
      → 内核初始化
        → init / PID 1 (systemd)
          → 启动系统服务（网络、打印……）
            → 启动登录管理器
              → 显示登录界面
```

**为什么 init 重要？**

- 它是所有其他进程的祖先
- 所有进程都是直接或间接由 init fork 出来的
- 如果 init 挂了，整个系统就挂了

---

## 6. 破坏性实验

```bash
# 1. 观察内核启动日志
dmesg | head -50

# 2. 查看 PID 1 的进程
cat /proc/1/status

# 3. 查看当前系统的进程树（简化版）
pstree | head -30
```

> **环境要求：** Linux 系统（Ubuntu/Debian/CentOS 等使用 systemd 的发行版）。`dmesg` 可能需要 root 权限（用 `sudo dmesg`）。

**验证标准（看到这些就算成功）：**

| 实验 | 你应该看到 | 说明 |
|------|-----------|------|
| `dmesg \| head -50` | 第一行包含 `Linux version`，方括号里有 `[0.000000]` 开头的时间戳 | 这是内核启动的第一条日志 |
| `cat /proc/1/status` | `Name:` 行显示 `systemd`（或 `init`），`Pid:` 行显示 `1` | PID 1 就是 init 进程 |
| `pstree \| head -30` | 最顶层是 `systemd`（或 `init`），下面分叉出多个子进程 | 所有进程都是 PID 1 的后代 |

## 7. 知识地图

```
电源按钮
    │
    ▼
BIOS/UEFI 自检
    │ ←── 硬件检测，住在主板 ROM 里
    ▼
Bootloader（GRUB）
    │ ←── 512 字节，负责加载内核
    ▼
Linux 内核
    │ ←── 管理 CPU、内存、外设
    ▼
init / systemd（PID 1）
    │ ←── 第一个用户进程，所有进程的祖先
    ▼
系统服务（network.target, sshd, ...）
    ▼
登录界面（getty, gdm, ...）
```

---

## 8. 设计决策总结

| 决策 | 原因 | 代价 |
|------|------|------|
| BIOS 住在 ROM 里 | 断电不丢数据，最早的上电入口 | 更新困难 |
| Bootloader 只有 512 字节 | 软盘扇区大小历史包袱 | 需要分阶段加载 |
| 内核是所有资源的管理者 | 统一管理，效率高 | 内核 bug 导致系统崩溃 |
| init 是第一个用户进程 | 需要一个"启动其他所有进程的进程" | init 挂了全挂 |
