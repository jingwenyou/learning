# CVSS漏洞评级 — 实战练习指南

> 本练习不需要靶场，主要使用 CVSS 计算器和你之前发现的漏洞。
> 目标：学会对任何漏洞准确打出 CVSS 分数，并能在漏洞报告中正确使用。

## 工具准备

```
CVSS 3.1 在线计算器（必备）：
  https://www.first.org/cvss/calculator/3.1

  4. 复制页面上的"Vector String"放进漏洞报告
```

## 练习1：给经典漏洞打分（理解指标含义）

### 步骤1.1：反射型 XSS（需用户点击链接）

```
场景：登录页面的 error 参数存在反射型 XSS，攻击者发链接给受害者，
     受害者点击后 Cookie 被发送到攻击者服务器。

逐项判断：

问：攻击者在哪里发起攻击？
  → 通过互联网发链接 → AV:N（网络）

问：攻击有什么特殊条件吗？
  → 发链接就行，无特殊条件 → AC:L（低）

问：攻击者需要有账号吗？
  → 不需要，任何人都能构造链接 → PR:N（无）

问：需要受害者主动操作吗？
  → 受害者必须点击链接 → UI:R（需要）

问：权限边界有没有扩大？
  → 攻击者只获取了受害者在当前网站的权限，没有扩大 → S:U（未改变）

问：机密性影响多大？
  → Cookie泄露 = 身份认证信息丢失 = C:H（高）

问：完整性影响多大？
  → 攻击者能以受害者身份操作 = I:H（高）

问：可用性影响多大？
  → 不影响服务正常运行 = A:N（无）

向量：CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:N

→ 在计算器中输入这些值
→ Expected 分数：约 8.8（高危）
→ 如果你的结果在 8.0-9.0 范围内，判断正确
```

### 步骤1.2：已认证的存储型 XSS（发帖植入脚本）

```
场景：论坛发帖功能可以提交 <script> 标签，其他用户浏览帖子时
     自动执行攻击者的脚本，可以窃取 Cookie 并劫持账号。

自己判断每个指标，然后和下面的答案对比：

你的答案：
  AV = ___   （提示：攻击者在哪里？在论坛上发帖）
  AC = ___   （有特殊条件吗？）
  PR = ___   （需要账号吗？）
  UI = ___   （需要受害者浏览帖子吗？）
  S  = ___   （权限边界扩大了吗？能劫持所有看帖人的账号？）
  C  = ___
  I  = ___
  A  = ___

参考答案：
  AV:N  — 互联网访问论坛
  AC:L  — 发帖即可，无特殊条件
  PR:L  — 需要一个普通用户账号才能发帖
  UI:R  — 需要其他用户浏览帖子
  S:C   — 能劫持任意看帖用户的账号（超出发帖者本身的权限）
  C:H   — Cookie泄露
  I:H   — 账号劫持可以做任何操作
  A:N   — 不影响可用性

向量：CVSS:3.1/AV:N/AC:L/PR:L/UI:R/S:C/C:H/I:H/A:N
Expected 分数：约 9.0（严重）

注意：为什么比反射型 XSS 还高？
  存储型XSS的S:C（Scope Changed），因为攻击者账号能影响其他用户
```

### 步骤1.3：SQL 注入（未认证，可读取数据库）

```
场景：商城首页的商品搜索接口存在SQL注入，无需登录即可利用，
     可以读取所有用户的邮箱、密码Hash、订单信息。

自己判断每个指标：

你的答案：
  AV = ___
  AC = ___
  PR = ___   （需要登录吗？）
  UI = ___
  S  = ___   （只读取数据，没有扩大到其他系统吗？）
  C  = ___   （能读取所有用户数据？）
  I  = ___   （能修改数据库？）
  A  = ___

参考答案：
  AV:N / AC:L / PR:N / UI:N / S:C / C:H / I:H / A:L

向量：CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:L
Expected 分数：约 9.8（严重）

为什么I:H？
  SQL注入通常可以 UNION SELECT，也可能 UPDATE/DELETE 数据
  评分时按最坏情况评估

为什么A:L不是N？
  大量查询可能拖慢数据库，有一定可用性影响
```

## 练习2：对同一漏洞评两种环境（理解环境影响）

```
场景：同一个未认证 SQL 注入漏洞，在两种环境下评分

环境A：内网测试服务器（只有内网访问）
  AV:A（相邻网络）/ AC:L / PR:N / UI:N / S:C / C:H / I:H / A:L
  → Expected 分数：约 8.6（高危）

环境B：公网生产服务器（所有人都能访问）
  AV:N（网络）/ AC:L / PR:N / UI:N / S:C / C:H / I:H / A:L
  → Expected 分数：约 9.8（严重）

体会：
  同一个漏洞，AV 从 A 改为 N，分数从 8.6 → 9.8
  这就是为什么公网服务器的漏洞优先级远高于内网服务器
```

## 练习3：给 DVWA 靶场漏洞打分（真实操作）

```
前提：DVWA 已部署并运行（bash labs/setup-labs.sh）
如果没有靶场，用练习1/2的场景代替即可

步骤：

1. 登录 DVWA → Security → 设置为 Low

2. 进入 SQL Injection 模块，输入 1' 触发报错
   → 确认漏洞存在

3. 现在用 CVSS 计算器给这个漏洞打分：

   DVWA 的 SQL 注入（Low 级别）的特征：
   - 能通过 HTTP 访问（AV?）
   - 只需发送 payload，无特殊条件（AC?）
   - 需要登录 DVWA 才能看到这个页面（PR?）
   - 不需要别人配合（UI?）
   - 能读取数据库里所有用户数据（S? C?）
   - 可能也能修改数据（I?）
   - 影响可用性吗（A?）

4. 打开计算器，填入你的判断，记录：
   - 向量字符串：___________
   - 分数：___________
   - 等级：___________

5. 把这个分数填到你的漏洞报告中（参考 Phase 5 漏洞报告模板）
```

## 练习4：对比不同漏洞的优先级（模拟真实工作场景）

```
假设你在一次安全测试中发现了以下 4 个漏洞，
按优先级排序，决定哪个最需要紧急修复：

漏洞A：服务器信息泄露（HTTP响应头暴露了服务器版本）
  攻击者可以知道你用的什么软件版本，但无法直接利用。
  AV:N / AC:L / PR:N / UI:N / S:U / C:L / I:N / A:N

漏洞B：DVWA XSS（反射型，需要用户点击）
  能窃取普通用户的 Cookie
  AV:N / AC:L / PR:N / UI:R / S:U / C:L / I:N / A:N

漏洞C：管理后台越权（普通用户能看到其他用户订单）
  登录后可以查看任意用户的订单
  AV:N / AC:L / PR:L / UI:N / S:U / C:H / I:N / A:N

漏洞D：未认证API接口（/api/admin/users 无需认证可访问）
  能获取所有用户的完整信息（含手机号、邮箱）
  AV:N / AC:L / PR:N / UI:N / S:U / C:H / I:N / A:N

步骤：
1. 对每个漏洞用计算器算出分数
2. 按分数从高到低排序
3. 对应等级是什么？
4. 哪个应该最先修复？

参考答案（不要看！先自己做）：
  漏洞D > 漏洞C > 漏洞B > 漏洞A
  漏洞D: 约7.5（高危）
  漏洞C: 约6.5（中危）
  漏洞B: 约4.3（中危）
  漏洞A: 约5.3（中危）
```

## 练习5：PortSwigger CVSS 场景

```
PortSwigger 的漏洞库里包含真实 CVE 的 CVSS 向量。

练习步骤：
1. 访问 https://portswigger.net/web-security（无需登录）
2. 进入任意一个漏洞类别（如 SQL Injection）
3. 阅读漏洞介绍
4. 自己给这个漏洞场景打 CVSS 分数
5. 搜索该漏洞的真实 CVE（如 CVE-2021-44228 Log4Shell）
6. 在 https://nvd.nist.gov/vuln/search 搜索 CVE 编号
7. 对比官方 CVSS 向量和你的判断

推荐练习的 CVE（都有官方CVSS评分可以对比）：
  CVE-2021-44228  Log4Shell（Log4j RCE）→ 官方 10.0
  CVE-2021-26855  Exchange ProxyLogon → 官方 9.8
  CVE-2022-22965  Spring4Shell → 官方 9.8
```
