# 04 CVSS漏洞评级 —— 给每个漏洞一个客观的分数

> 你发现了漏洞，下一步是告诉别人它有多严重。
> CVSS（通用漏洞评分系统）是安全行业评估漏洞严重程度的**标准语言**。
> 学会用它，你就能和开发、安全团队、管理层用同一套标准沟通。

---

## 一、为什么需要CVSS

```
不用CVSS的对话：
  "这个漏洞很严重！"
  "多严重？"
  "很严重！"

用CVSS的对话：
  "这个漏洞CVSS评分9.8，严重的。"
  "那比评分7.5的如何？"
  "严重多了，需要立即修复。"

CVSS提供了：
- 客观的评分标准（不是主观感觉）
- 可对比的量化指标（不同漏洞之间可以比较）
- 行业通用的语言（安全公司、监管机构都用这个）
```

---

## 二、CVSS 3.1 评分体系

### 2.1 三个维度

```
CVSS分数由三个维度组成：

[1] 基本评分 (Base Score) —— 漏洞本身的固有属性
    不随时间、环境变化，永远不变
    例子：SQL注入 → 可以获取数据 → 影响很严重

[2] 时效评分 (Temporal Score) —— 随时间变化的因素
    例如：漏洞利用代码是否公开、是否有修复方案
    随时间推移，分数可能降低

[3] 环境评分 (Environmental Score) —— 取决于你的环境
    例如：你的系统在内网还是公网
    同一漏洞在不同环境下严重程度不同

实际使用中：
  基本评分（0-10）是必填的
  时效和环境评分是可选的（用于更精确的评估）
  通常说的"CVSS评分"指的是基本评分
```

### 2.2 基本评分的6个度量指标

```
公式：CVSS = 6.42 × [ISC + (ISCa × IR + IACR × IRA + IAC × IEX + EC × IEV + RL × RV + RC × RCc)]

但你不需要记公式。只需要理解每个指标的含义。

┌─────────────────────────────────────────────────────────┐
│                  六 大 度 量 指 标                        │
│                                                         │
│  攻击向量(AV)    → 攻击者从哪里发起攻击？                │
│  攻击复杂度(AC)  → 攻击有多难？                         │
│  所需权限(PR)    → 需要什么权限？                       │
│  用户交互(UI)    → 需要受害者配合吗？                    │
│  影响范围(C)      → 能影响哪些数据/系统？Confidentiality  │
│  影响范围(I)      → Integrity（数据完整性）              │
│  影响范围(A)      → Availability（可用性）               │
└─────────────────────────────────────────────────────────┘
```

### 2.3 每个指标的详细说明

```
═══════════════════════════════════════════════════════════
                        攻击向量 (Attack Vector, AV)
═══════════════════════════════════════════════════════════
攻击者从哪里发起攻击？

网络 (N)      6.84分 — 通过互联网远程攻击，最严重
              例：直接发HTTP请求打掉你的服务器
              CVSS向量: AV:N

相邻 (A)      4.94分 — 需要在同一个网络（如WiFi、蓝牙）
              例：同局域网的其他机器
              CVSS向量: AV:A

本地 (L)      2.24分 — 需要物理接触或本地账号
              例：先登录服务器才能利用
              CVSS向量: AV:L

物理 (P)      0.27分 — 必须物理接触
              例：插U盘、放个伪基站
              CVSS向量: AV:P

理解窍门：攻击者能躺在家里的沙发上打，就是N。

═══════════════════════════════════════════════════════════
                       攻击复杂度 (Attack Complexity, AC)
═══════════════════════════════════════════════════════════
攻击的难度除了网络位置，还需要什么条件？

低 (L)        0.77分 — 没有特殊条件，正常网络请求即可
              例：SQL注入，只要发送payload就能成功
              CVSS向量: AC:L

高 (H)        0.62分 — 需要特殊条件：时序、特定的系统状态等
              例：需要攻击者碰巧遇到用户登录的瞬间
              CVSS向量: AC:H

═══════════════════════════════════════════════════════════
                        所需权限 (Privileges Required, PR)
═══════════════════════════════════════════════════════════
攻击者在开始攻击前，需要有什么权限？

无 (N)        0.85分 — 匿名用户即可，不需要登录
              例：未认证的SQL注入、反射型XSS
              CVSS向量: PR:N

低 (L)        0.62分 — 需要普通用户权限
              例：普通用户能越权看其他用户数据
              CVSS向量: PR:L

高 (H)        0.27分 — 需要管理员权限
              例：在已有admin账号的情况下再提权
              CVSS向量: PR:H

注意：如果AV是N（网络可达）且PR是N（无需权限），这个漏洞通常评分很高。

═══════════════════════════════════════════════════════════
                        用户交互 (User Interaction, UI)
═══════════════════════════════════════════════════════════
攻击是否需要受害者在不知情的情况下做点什么？

无 (N)        0.85分 — 不需要受害者配合
              例：服务器自动处理的漏洞（SQL注入）
              CVSS向量: UI:N

需要 (R)      0.62分 — 需要受害者主动配合（如点击链接）
              例：存储型XSS需要别人浏览你的帖子
              CVSS向量: UI:R

═══════════════════════════════════════════════════════════
                   影响范围 — CIA三元组 (C/I/A)
═══════════════════════════════════════════════════════════
漏洞利用成功后，能造成多大影响？

这三个指标比较特殊，取决于漏洞是否能够突破"权限边界"：

不变 (N)      0分   — 不影响其他组件，漏洞影响只在自己的权限范围内
              例：普通用户越权看其他用户的同级别数据

改变 (C/H)    视情况
              — Scope Changed（权限边界被突破）→ 影响范围扩大
              — Scope Unchanged（权限边界未被突破）→ 影响范围受限

              怎么判断Scope Changed？
              漏洞利用后，攻击者能做超出自己原本权限的事。

举例：
  普通用户发现一个漏洞，能直接看到管理员的数据
  → 自己只有PR:L（低权限），但能看到PR:H（高权限）的数据
  → Scope Changed → 评分大幅提升
```

### 2.4 CIA影响的评分

```
Scope Unchanged（权限边界未被突破）：

  C/I/A三个指标各自独立评分：
  高 (H): 0.56
  低 (L): 0.22
  无 (N): 0

  综合影响 = 1 - [(1-C) × (1-I) × (1-A)]

  举例：C=H, I=H, A=N
  = 1 - [(1-0.56) × (1-0.56) × (1-0)]
  = 1 - [0.44 × 0.44 × 1]
  = 1 - 0.1936
  ≈ 0.81  → 这个值用于最终计算

Scope Changed（权限边界被突破）：

  C/I/A的计算值会乘以一个系数（通常1.5左右）
  Scope Changed通常比Unchanged高2-3分
```

## 三、CVSS评分计算示例

### 示例1：未认证的反射型XSS

```
场景：攻击者发一个链接给受害者，受害者点击后Cookie被盗

逐项分析：

AV   = N（网络）      — 攻击者通过互联网发送链接
AC   = L（低）        — 不需要特殊条件，点链接就行
PR   = N（无）        — 不需要登录账号
UI   = R（需要）      — 受害者必须点击链接
S    = U（未改变）    — 权限边界没有扩大，只是偷自己的Cookie
C    = H（高）        — Cookie泄露 = 认证信息丢失 = 高度机密性影响
I    = H（高）        — 攻击者可以冒充受害者操作 = 高度完整性影响
A    = N（无）        — 不影响服务可用性

向量：CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:N

基本评分：约6.1（中-高）
解释：这是一个中高级别的漏洞，需要受害者配合，但影响严重
```

### 示例2：已认证的SQL注入（可UNION读取数据）

```
场景：登录后的搜索功能存在SQL注入，可以读取数据库中所有表

逐项分析：

AV   = N（网络）      — 互联网远程
AC   = L（低）        — 发送SQL payload即可
PR   = L（低）        — 需要一个登录账号（普通用户即可）
UI   = N（无）        — 不需要受害者配合，自动化脚本可完成
S    = C（改变）      — 能读取超出自己权限的数据（Scope Changed）
C    = H（高）        — 能读取所有用户的密码等敏感数据
I    = H（高）        — 能修改数据库（通过UNION UPDATE等）
A    = L（低）        — 读取数据对可用性影响有限

向量：CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:L

基本评分：约8.8（高危）
解释：远程无需交互，但需要低权限账号 → 高危
```

### 示例3：内网服务器的远程命令执行

```
场景：内网监控系统开放了一个未授权API，攻击者可执行任意命令

逐项分析：

AV   = A（相邻）      — 需要在内网中，不能从互联网直接访问
AC   = L（低）        — 发送POST请求即可，无特殊条件
PR   = N（无）        — 不需要任何账号
UI   = N（无）        — 不需要受害者操作
S    = C（改变）      — 命令执行直接突破权限边界
C    = H（高）        — 能读取服务器上所有数据
I    = H（高）        — 能修改服务器上的任何东西
A    = H（高）        — 能直接关掉服务器

向量：CVSS:3.1/AV:A/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:H

基本评分：约8.6（高危）
解释：虽然在内网，但无认证+命令执行 → 严重
```

## 四、CVSS评分等级

```
CVSS分数范围和对应等级：

9.0 - 10.0   严重 (Critical)    ← 最优先处理
              例：远程无认证RCE、严重敏感数据泄露

7.0 - 8.9    高危 (High)
              例：未认证SQL注入、管理员越权、存储型XSS

4.0 - 6.9    中危 (Medium)
              例：反射型XSS、需要交互的CSRF、认证后的越权

0.1 - 3.9    低危 (Low)
              例：信息泄露、安全头缺失、Self-XSS

0.0          无危 (None)
              例：不是漏洞的安全建议
```

## 五、CVSS在线计算器

```
实际工作中，不需要手动算。用在线工具：

1. FIRST.org 官方CVSS 3.1计算器
   https://www.first.org/cvss/calculator/3.1

2. Bugcrowd的CVSS计算器（更友好）
   https://bugcrowd.com/cvss-search

3. 中国的：绿盟、启明的漏洞管理平台也有CVSS计算器

使用方法：
  1. 根据漏洞特征选择每个指标
  2. 系统自动算出分数
  3. 导出向量字符串（如 CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:N）
  4. 在漏洞报告中附上向量字符串，便于他人验证
```

## 六、CVSS vs 其他评级体系

```
OWASP风险评级方法（替代CVSS，适合业务评审）：

OWASP用概率 × 影响 来评估：

概率评估因素：
  - 攻击者技能水平（1-3分）
  - 漏洞利用难度（1-4分）
  - 认证需求（1-3分）
  - 利用普遍性（0-5分）

影响评估因素：
  - 技术影响（机密性、完整性、可用性）
  - 业务影响（财务损失、声誉损失……）

两者相乘得到风险等级：
  高-中-低

CVSS更技术化，适合安全工程师内部沟通
OWASP风险评级更适合向业务方（PM、管理层）汇报
```

## 七、动手实践

### 练习1：给自己发现的漏洞打分

```
回到你之前在DVWA/其他靶场发现的漏洞，
用CVSS计算器给它打分：

1. 登录DVWA，SQL注入Low级别
2. 问自己6个问题：
   - 攻击者在哪里？→ N（网络）
   - 攻击难不难？→ L（低，发个'就行）
   - 需要什么权限？→ N（不需要，URL参数就能注入）
   - 需要受害者配合吗？→ N（自动化脚本即可）
   - 影响范围扩大了吗？→ C（改变了，能读所有数据）
   - CIA影响多大？

3. 用计算器验证你的判断
4. 把分数写进你的漏洞报告
```

### 练习2：对同一漏洞评两种级别

```
同一个SQL注入漏洞：
  场景A：公司的内网测试环境（AV=A，相邻网络）
  场景B：公网生产环境（AV=N，网络可达）

对比两个评分，理解为什么同一漏洞在不同环境下优先级不同。
```
