# 阶段 2：进程

## 上节课回顾

上节课看到计算机从开机到登录界面的接力过程：BIOS → Bootloader → 内核 → init (PID 1) → 系统服务。init 是第一个用户进程，它会启动更多进程。**这节课问：操作系统怎么"跟踪"一个运行中的程序？**

---

## 1. 什么是进程？

**大白话：**

程序是存在磁盘上的文件（比如 `/bin/ls`），它是"死的"代码。

进程是程序运行起来后的"实例"，是"活的"。

一个程序可以同时跑多个进程。比如你开两个 terminal，都跑 `sleep 100`，这是两个独立的进程，虽然它们执行的是同一个程序。

```
程序（磁盘上的文件）     进程（内存中运行的实例）
┌─────────────┐         ┌─────────────┐
│   /bin/sleep│   运行   │  sleep PID 12345 │
│   (代码+数据)│ ──────→ │  sleep PID 12346 │
└─────────────┘         └─────────────┘
                                  ↑
                         两个独立的进程
```

**设计决策：**

- **Problem**：需要区分"代码本身"和"正在运行的实例"，因为多个实例可以同时跑
- **Solution**：程序 = 静态代码，进程 = 动态运行实例
- **Tradeoffs**：需要额外的内存来维护进程状态
- **Limitation**：进程创建有开销（fork 的成本）

---

## 2. task_struct：进程的"身份证"

**大白话：**

操作系统需要跟踪每一个进程。怎么跟踪？给每个进程发一张"身份证"。

在 Linux 里，这个身份证叫 `task_struct`（任务结构体）。它存在内核的内存里，里面写着：

| 字段 | 含义 |
|------|------|
| pid | 进程 ID，唯一标识 |
| state | 进程状态（运行、睡眠、僵死……） |
| ... | 还有几十个字段（内存、文件、信号等），后面阶段会逐个学到 |

> 完整字段参考（现阶段了解即可，不用记）：
>
> | 字段 | 含义 | 对应阶段 |
> |------|------|---------|
> | se.vruntime | 虚拟运行时间（CFS 调度器用，越小越优先） | 阶段 7 |
> | policy | 调度策略 | 阶段 7 |
> | mm | 内存管理信息（指向地址空间） | 阶段 3 |
> | files | 文件描述符表 | 阶段 4 |
> | signal | 信号处理信息 | 阶段 6 |

**设计决策：**

- **Problem**：内核需要管理很多进程，每个进程有大量状态要记录
- **Solution**：用双向链表把所有 `task_struct` 连起来（用于遍历全部进程），同时用 **PID 哈希表**（pidhash）做 O(1) 的按 PID 查找
- **Tradeoffs**：链表遍历是 O(n)，但只在需要扫描全部进程时才用；按 PID 查找走哈希表，是 O(1)
- **Limitation**：task_struct 本身较大（几 KB），进程数很多时内存开销不小

**历史渊源：**

最早的 Unix 用 `proc` 结构体，后来 Linux 改成了 `task_struct`，但思想是一样的。Linux 的 `task_struct` 比早期 Unix 大很多，因为功能多了。

**源码位置：**

`include/linux/sched.h` 里定义了 `task_struct`（L2）

## 3. 进程状态

**大白话：**

每个进程在任意时刻处于某种"状态"：

| 状态 | 符号 | 什么意思 |
|------|------|---------|
| 运行 | R (Running) | 正在 CPU 上跑，或者在就绪队列里等着跑 |
| 睡眠 | S (Sleeping) | 在等某个事件，比如等 I/O、等定时器 |
| 磁盘睡眠 | D (Disk Sleeping) | 不可中断的睡眠，等 I/O，不能被信号唤醒 |
| 停止 | T (Stopped) | 被暂停了（收到 SIGSTOP 信号） |
| 僵死 | Z (Zombie) | 进程死了，但父进程还没收尸 |

**重点：僵死状态（Zombie）**

进程退出时，它的退出码需要被父进程读取。父进程可能还没准备好读取，所以内核先保留这个状态，等父进程来收。

```
进程 A（父进程）
  → fork() 创建进程 B
  → B 死了，变成 Z（僵死）
  → A 没调用 wait()，B 就一直僵死着
  → A 调用 wait()，B 的尸体被回收
```

**为什么有这个状态？**

因为进程退出时，它的退出码（exit code）是给父进程用的。父进程可能还在干别的事，所以内核先保留这个状态，等父进程来读取。

## 4. 进程创建：fork 和 exec

**大白话：**

Linux 创建进程有两种方式：

**fork（分叉）：** 复制当前进程，创建一个"分身"

大白话理解：
```
想象你在看一本书，看到第 50 页。
fork() 就像把你和这本书"复制"了一份。
现在有两个"你"，都拿着同一本书，都翻到第 50 页。
唯一的区别：原来的你手里多了一张纸条写着"副本编号"，副本手里的纸条写着"0"。
两个人根据纸条上的数字，决定接下来干什么。
```

翻译成代码：
- 调用 fork 后，父进程和子进程几乎一模一样
- 唯一的区别：fork 的返回值不同（父进程拿到子进程 PID，子进程拿到 0）
- 之后的执行路径由返回值决定

```c
pid_t pid = fork();

if (pid == 0) {
    // 我是子进程
    printf("我是子进程\n");
} else {
    // 我是父进程
    printf("父进程，子进程 PID 是 %d\n", pid);
}
```

**exec（替换）：** 把当前进程"变身"成另一个程序
- 调用 exec 后，当前进程的代码段被新程序替换
- PID 不变，但执行的程序变了

```c
// 在子进程里执行 ls
execlp("ls", "ls", "-l", NULL);
// 如果 exec 成功，这行之后的代码不会执行
// 因为当前进程已经变成 ls 了
```

**通常的用法（fork-exec 模式）：**

```
fork() 创建一个子进程
  → 子进程调用 exec() 变成另一个程序
  → 父进程调用 wait() 等子进程结束
```

**设计决策：**

- **fork 为什么需要？** 需要一个进程创建新进程，fork 是最直接的方式
- **fork 为什么不直接创建全新进程？** 因为子进程通常要执行新程序，复制父进程的状态再做替换更灵活
- **Tradeoffs**：fork 的代价是复制整个地址空间（但用了 copy-on-write 优化，见下文）
- **Limitation**：fork 不能传递文件描述符给完全不相关的进程

**Copy-on-Write（写时复制）：**

fork 并不会真的立刻复制所有内存。实际做法是：

```
fork() 刚执行完：
┌──────────────┐       ┌──────────────┐
│  父进程       │       │  子进程       │
│  页表 ───────┼──┐    │  页表 ───────┼──┐
└──────────────┘  │    └──────────────┘  │
                  │                      │
                  └──────┬───────────────┘
                         ▼
                  ┌──────────────┐
                  │  同一块物理内存 │  ← 两个进程共享，标记为"只读"
                  └──────────────┘

当某个进程写入时：
                  ┌──────────────┐
  父进程页表 ────→│  原来的内存    │  ← 没改的一方继续用原来的
                  └──────────────┘
                  ┌──────────────┐
  子进程页表 ────→│  复制出的新页  │  ← 改了的一方拿到新副本
                  └──────────────┘
```

所以 fork 的真实开销很小——只复制页表（几 KB），不复制实际内存（可能几百 MB）。只有当某一方真的写入时，才复制那一页（4KB）。

## 4.1 常见疑问：fork() 到底是复制什么？

**问：fork() 后是"复制 main 函数"吗？**

**答：** 不是复制 main，是**复制整个进程**。

fork() 复制的是：
- 整个内存空间（代码段、数据段、堆、栈）
- 所有寄存器的值
- 打开的文件描述符
- 当前执行到了哪一行代码

复制完之后，两个进程都从 fork() 之后的下一行代码继续执行。

```
fork() 之前：
┌──────────────────┐
│  进程            │
│  代码: main()    │
│  第 1 行         │
│  ...             │
│  第 5 行: fork() ← CPU 在这里
│  第 6 行         │
│  ...             │
└──────────────────┘

fork() 之后（复制）：
┌──────────────────┐       ┌──────────────────┐
│  父进程          │       │  子进程          │
│  第 5 行返回 123 │       │  第 5 行返回 0   │
│  第 6 行继续执行 │       │  第 6 行继续执行 │ ← 和父进程执行位置一样
│  ...             │       │  ...             │
└──────────────────┘       └──────────────────┘
```

**问：fork() 调用一次返回两次，怎么做到的？**

**答：** 不是"调用两次"，是**返回值被两个进程分别读取**。

- 内核在 fork 返回时，已经创建了一个新的进程
- 父进程拿到子进程的 PID (返回值 = 子进程 PID)
- 子进程拿到 0（返回值 = 0）
- 两个进程继续执行

**问：子进程里 "父进程的 PID" 打印了两遍是怎么回事？**

**答：** 代码写错了，正确代码只有一次打印。正确代码示例：

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        // 我是子进程
        printf("子进程: 我的 PID 是 %d\n", getpid());
        printf("子进程: fork 返回值是 %d\n", pid);
        printf("子进程: 父进程的 PID 是 %d\n", getppid());
        return 0;
    } else {
        // 我是父进程
        printf("父进程: fork 返回值是 %d\n", pid);
        printf("父进程: 我的 PID 是 %d\n", getpid());
        waitpid(pid, NULL, 0);
    }

    return 0;
}
```

**验证：**
```bash
gcc test_fork.c -o test_fork && ./test_fork
```

> ⚠️ **常见问题：`test_fork: command not found`**
>
> 执行编译后的程序必须加 `./` 前缀，因为当前目录（`.`）不在 PATH 环境变量里。`./` 表示"在当前目录下找这个文件"。
>
> | 错误写法 | 正确写法 |
> |----------|----------|
> | `test_fork` | `./test_fork` |
> | `test_fork &` | `./test_fork &` |

## 4.2 补充：线程是什么？

**大白话：**

线程是"轻量级进程"。多个线程共享同一个进程的内存空间，但各自有独立的执行流（各跑各的代码）。

```
进程（独立的地址空间）          线程（共享地址空间）
┌────────────┐                ┌────────────────────┐
│ 进程 A     │                │ 进程 A             │
│ 自己的内存  │                │ 共享的内存          │
└────────────┘                │  ├── 线程 1（独立栈）│
┌────────────┐                │  ├── 线程 2（独立栈）│
│ 进程 B     │                │  └── 线程 3（独立栈）│
│ 自己的内存  │                └────────────────────┘
└────────────┘
```

**Linux 的实现：** 在 Linux 里，线程和进程用的是同一个 `task_struct`。创建线程用 `clone()` 系统调用，和 `fork()` 的区别只是参数不同——`clone()` 可以指定"共享父进程的内存空间"。所以在 Linux 内核看来，线程就是共享了地址空间的进程。

> 这个概念现阶段了解即可，阶段 8（同步原语）会深入讨论多线程的问题。

## 5. 进程调度基础

**大白话：**

多个进程同时"要跑"，但 CPU 只有几个。调度器决定谁先跑、谁后跑、谁跑多久。

调度器工作的基础是**定时器中断**（阶段 5 会详细讲）。每隔一段时间（比如 1ms），定时器会打断当前进程，把 CPU 控制权交给调度器，调度器决定让谁继续跑。

**典型数值：**

| 操作 | 典型耗时 |
|------|---------|
| 进程切换（context switch） | 1-5 μs |
| 创建新进程（fork） | 50-100 μs |
| 进程退出（exit） | 10-20 μs |

## 6. 破坏性实验

> **环境要求：** Linux 系统（Ubuntu/Debian/CentOS 等），部分实验需要 root 权限（`sudo`）。需要安装 `gcc`（用于编译 C 代码实验）和 `python3`。

### 实验 1：观察进程基本信息

```bash
# 运行一个程序，观察其 PID
sleep 100 &
echo "sleep 进程的 PID: $!"

# 观察这个进程的状态
ps aux | grep sleep

# 观察进程树
pstree

# 观察当前 shell 的进程信息
cat /proc/$$/status | grep -E "Name|Pid|State"

# ⚠️ 实验结束后清理：
kill %1
```

**验证标准：**

| 命令 | 你应该看到 | 说明 |
|------|-----------|------|
| `ps aux \| grep sleep` | 有一行包含 `sleep 100`，STAT 列显示 `S` | S = 睡眠状态，因为 sleep 在等定时器 |
| `pstree` | 树状结构，顶部是 `systemd`，你的 shell 和 sleep 在其中某个分支 | 所有进程都是 PID 1 的后代 |
| `cat /proc/$$/status` | `Name: bash`（或 zsh），`State: S (sleeping)` | 当前 shell 在等你输入命令，所以也是睡眠 |

### 实验 2：观察 fork — 编译运行 C 代码

把以下代码保存为 `test_fork.c`，编译运行：

```bash
cat > /tmp/test_fork.c << 'EOF'
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        printf("子进程: 我的 PID 是 %d\n", getpid());
        printf("子进程: fork 返回值是 %d\n", pid);
        printf("子进程: 父进程的 PID 是 %d\n", getppid());
        return 0;
    } else {
        printf("父进程: fork 返回值是 %d（= 子进程 PID）\n", pid);
        printf("父进程: 我的 PID 是 %d\n", getpid());
        waitpid(pid, NULL, 0);
        printf("父进程: 子进程已退出，已回收\n");
    }

    return 0;
}
EOF
gcc /tmp/test_fork.c -o /tmp/test_fork && /tmp/test_fork
```

**验证标准：**
- 输出 5 行（父进程 2 行 + 子进程 3 行 + 回收 1 行）
- 父进程的 `fork 返回值` = 子进程的 `我的 PID`
- 子进程的 `fork 返回值` = 0

### 实验 3：观察 exec — fork 后替换成新程序

```bash
cat > /tmp/test_exec.c << 'EOF'
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        printf("子进程: 我是 PID %d，马上变身成 ls\n", getpid());
        execlp("ls", "ls", "-l", "/tmp/test_fork.c", NULL);
        // 如果 exec 成功，下面这行不会执行
        //
        // 为什么有两个 "ls"？
        //   第一个 "ls"：在 PATH 中搜索的程序名
        //   第二个 "ls"：argv[0]，程序自己看到的名字
        //   (char *)0 或 NULL：参数列表结束标记
        printf("这行不应该出现\n");
    } else {
        waitpid(pid, NULL, 0);
        printf("父进程: 子进程（已变身成 ls）执行完毕\n");
    }

    return 0;
}
EOF
gcc /tmp/test_exec.c -o /tmp/test_exec && /tmp/test_exec
```

**验证标准：**
- 子进程打印 "马上变身成 ls" 后，直接显示 `ls -l` 的输出
- "这行不应该出现" 确实没出现（因为 exec 替换了整个程序）
- 父进程最后打印 "执行完毕"

### 实验 4：制造僵尸进程

```bash
# 步骤 1：运行一个父进程，它 fork 子进程但故意不调用 wait()
python3 -c "
import os, time, signal
pid = os.fork()
if pid == 0:
    # 子进程：立刻退出
    os._exit(0)
else:
    # 父进程：不回收子进程，等 60 秒
    print(f'父进程 PID: {os.getpid()}')
    print(f'子进程 PID: {pid}（应该变成僵尸）')
    print('现在在另一个终端运行: ps aux | grep Z')
    time.sleep(60)
" &

# 步骤 2：等 2 秒让子进程退出，然后观察
sleep 2
ps aux | grep Z

# ⚠️ 实验结束后清理（kill 父进程，僵尸进程自动被 init 回收）：
# 注意：%1 是 bash 作业号（job number），不是 PID
#   %1  = 第一个后台作业（实验中刚启动的 python3 父进程）
#   kill %1  = 杀掉这个后台作业的父进程
kill %1
```

**验证标准：**
- `ps aux | grep Z` 能看到一行 STAT 列为 `Z+` 的进程，CMD 列显示 `[python3] <defunct>`
- kill 掉父进程后，再执行 `ps aux | grep Z`，僵尸进程消失（被 init 回收了）

```
阶段 1（计算机启动）
    │
    └── init / systemd（PID 1）─────────────┐
                                            │
    ┌───────────────────────────────────────┘
    │
    ▼
进程
    │

    ├── exec() 替换成新程序

    ├── 进程退出后变成僵死（Z），等待父进程 wait() 回收
    │
    └── 进程状态：R / S / D / T / Z
```

## 下节课预告

**阶段 3：内存**

每个进程有 task_struct，里面有"内存指针"。进程运行时需要内存。**下节课问：程序用的地址和真实物理地址是什么关系？**
